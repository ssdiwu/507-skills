用户明确说“用 TDD 做 / 先写测试 / 红绿重构 / test-first”时，按红绿重构推进。

## 核心循环

```text
RED: 写一个行为测试 → 确认因为预期原因失败
GREEN: 写最小实现 → 确认通过
REFACTOR: 全绿后再重构
```

反模式是水平切片（先写所有测试再写所有实现）。正确做法是 **vertical slice（垂直切片）/ tracer bullet（示踪弹）**：一次只写一个行为测试，只写够当前测试通过的代码；每个新测试响应上一轮实现学到的东西。

## TDD 流程

1. **开始前确认**：公共接口长什么样？最重要行为是什么？哪些边界先不测？只列行为，不列实现步骤。
2. **第一颗示踪弹**：写打通主路径的行为测试，确认因为预期原因失败；编译错误、fixture 错误、测试环境错误不算有效 RED。
3. **最小实现**：写刚好让当前测试通过的代码，不预判未来功能。
4. **增量循环**：每轮只加一个行为测试 → 确认失败 → 最小实现 → 确认通过。
5. **绿灯下重构**：全部通过后再重构，绝不在 RED 时重构，每次重构后跑测试。
6. **收口**：说明完成了哪些行为、跑了哪些测试、还有哪些边界未覆盖。

测试构造纪律：接缝坏了（不得不穿透很多内部结构 / 要伪造巨大对象才能跑）先怀疑模块形状或公开接口设计有问题，而不是硬构造；partial、故意错类型这类放宽写法只在测试里，不进生产代码。

## 每轮检查清单

```text
[ ] 测试描述行为，不描述实现
[ ] 测试只走公共接口
[ ] 测试会在内部重构后继续通过
[ ] RED 是因为预期行为缺失而失败，不是因为测试写错或环境坏了
[ ] 当前代码只是让这个测试通过的最小代码
[ ] 没有提前加未来功能
```

## 好测试 / 坏测试

好测试：

- “用户能用有效购物车结账”
- “无权限用户调用接口得到 403”

坏测试：

- “调用了内部函数 X 三次”
- “createUser 后直接 SQL 查表，而项目本来有公开读取接口”

## 输出

最后用中文简要说明：使用了 TDD 模式；新增/修改了哪些测试；覆盖了哪些行为；跑了哪些命令、结果是什么；未覆盖的边界和原因。

## 完成后

红绿重构走完后不等于收尾。继续走 `/review` 审查改动质量与文档同步，再走 `/commit` 提交。如果实现暴露了新行为或新术语，提醒用户同步术语表或 doc。
