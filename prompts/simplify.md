## 目标

在**保持功能不变**的前提下降低复杂度。它适合最近改动后的清理、局部重构、小步简化；不适合重新设计系统架构。

如果发现问题已经超出局部简化（例如模块形状不对、没有正确测试接缝、需要重新划分职责），先停止扩大，建议切到 `507-inspection` 或 `507-workorder`。

## 范围确认

如果补充内容里指定了文件或路径，只审查这些文件或路径。若没有指定，先运行：

```bash
git diff --name-status HEAD
```

根据输出列出将要审查的文件；如果没有当前改动，再问用户是否改为审查最近三次提交，不要自动扩大范围。

如果当前目录不是 Git 仓库或命令失败，不要自行全盘扫描；请用户指定文件 / 目录范围，并推荐一个最小范围。

## 原则

1. **保持行为不变**：不改业务行为、不新增功能、不修改 public API（公开接口）。
2. **小步重构**：每一步都尽量小，且应该让代码保持可运行；不要一口气大搬家。
3. **遵循项目规范**：先读相关 `AGENTS.md`、目标目录及上下级 `README.md`。
4. **价值明确**：只做能明确降低复杂度、减少重复、改善命名、缩短过长函数、删除无用代码的改动。
5. **不过度抽象**：没有第三次重复，不要提前抽象；一个 adapter（适配器）不要提前造 seam（接缝）。
6. **删除测试**：判断一个抽象是否值得留，想象删掉它——复杂度消失就是透传，复杂度在多个调用处重现才值得留。
7. **范围受控**：只改确认范围内文件；必须改范围外文件时，先说明原因并请求确认。
8. **安全优先**：不硬编码密钥，不放宽校验，不隐藏错误。

## 流程

1. **确认范围**：列出将审查的文件和原因。
2. **读上下文**：读相关文档和代码，理解当前职责边界。
3. **找候选**：列出可简化点，优先级从“低风险高收益”开始。
4. **小步修改**：一次只做一个明确简化；避免大规模重排。
5. **保持绿色**：每个重要步骤后运行最小可行验证；至少在结束前跑相关测试、类型检查或命令。
6. **收口**：说明做了哪些简化、为什么安全、验证结果是什么。

## 常见候选

- 重复代码 → 抽出更小的函数 / 常量（仅当重复足够真实）
- 过长函数 → 拆成私有 helper，但测试仍走公共接口
- 透传模块 → 删除或合并
- 命名含糊 → 用项目术语改名
- 过早抽象 → 内联回调用处
- 无用注释 / 死代码 → 删除
- 新代码暴露出的旧代码问题 → 记录并说明是否超出本次范围

## 输出

最后用中文总结：

- 修改了哪些文件
- 每个改动降低了什么复杂度
- 为什么不改变行为
- 运行了什么验证
- 是否有超出本次范围、建议后续走 `507-inspection` 的问题

## 红线

- 不新增功能。
- 不改变外部行为。
- 不顺手做架构重设。
- 不修改范围外文件，除非先获确认。
- 不把“更优雅”当理由；必须能说明真实收益。

## 完成后

简化验证通过后，不等于收尾。继续走 `/review` 审查改动质量与文档同步，再走 `/commit` 提交。如果重构改变了模块职责或边界，提醒用户同步分层 README 或 doc。
