## 目标

本 prompt 用于**测试本身是主要任务**的场景：补测试、跑测试、修失败测试、缩小失败范围、提升关键行为覆盖率。

## 原则

1. **测试看行为**：优先通过公共接口验证调用者关心的行为，不测私有方法和内部实现细节。
2. **用现有框架**：沿用项目已有测试框架、测试命令和命名习惯，不引入新测试栈。
3. **mock 只放系统边界**：第三方 API、数据库、时间/随机数、文件系统可 mock；不要 mock 自己的内部类/模块/协作者。
4. **断言具体症状**：断言明确行为、输出、状态或错误；不要只写“没有报错”。
5. **测试应抗重构**：内部重命名或拆函数不应导致测试坏；如果坏了，说明测试耦合实现。
6. **最小有效范围**：先跑最相关的测试用例/文件/包，再按需扩大到完整套件。
7. **失败测试先当信号**：先判断失败代表真实 bug、spec 变化还是测试过时，不要上来就改断言。
8. **接缝坏了先怀疑形状**：如果为了测试不得不穿透很多内部结构、或要伪造巨大对象才能跑起来，优先怀疑模块形状 / 公开接口设计有问题，而不是先想办法硬构造。
9. **放宽构造只在测试里**：partial、故意错类型这类放宽写法只允许出现在测试，不进生产代码；跟随项目现状，不为技巧强装依赖。

## 流程

1. **识别测试目标**：补测试、跑测试、修失败测试，还是缩小失败范围？
2. **读失败输出**：修失败测试时，先读完整错误、定位首个真实失败，确认它是否因为预期原因失败。
3. **读项目惯例**：先看相邻测试文件、测试命令、README/AGENTS.md 里的测试约定。
4. **选择测试接缝**：选能覆盖真实行为的最高公开接缝；避免绕过接口直接查内部状态。
5. **写或调整测试**：测试名描述行为场景；优先正常路径、关键边界、错误情况，不追求全边界。
6. **运行最小测试集**：先跑单个用例/文件/包；通过后再决定是否扩大范围。
7. **报告证据**：说明跑了什么命令、结果如何、失败时给出关键错误和下一步。

## 好测试 / 坏测试

好测试：

- “用户能用有效购物车结账”
- “无权限用户调用接口得到 403”

坏测试：

- “调用了内部函数 X 三次”
- “createUser 后直接 SQL 查表，而项目本来有公开读取接口”

## 输出

最后用中文简要说明：

- 新增/修改了哪些测试
- 覆盖了哪些行为
- 跑了哪些命令，结果是什么
- 未覆盖的边界和原因
