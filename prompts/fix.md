遵循以下原则：

1. **最小改动** — 只修改修复问题所必需的部分
2. **根因修复** — 修复根本原因，而不是症状
3. **无回归** — 确保现有行为保持不变
4. **先建反馈环** — 先做一个可重复的 pass/fail 信号，再开始改代码
5. **解释** — 简要说明 bug 的原因和修复原理

## 错误分析步骤

遇到报错 / 崩溃 / 异常时，按这个顺序定位：

1. **读错** — 完整读 stack trace（堆栈跟踪）和错误信息，不要只看最后一行
2. **定位** — 找到报错源文件和行号
3. **读上下文** — 读报错点前后代码，理解它在做什么
4. **复现** — 确认最小复现步骤（命令 / 输入 / 操作）
5. **反馈环** — 把复现固化成失败测试、curl、CLI 重放、headless 操作或最小脚本；没有确定信号就不要盯着代码空猜
6. **根因** — 沿数据流回溯，找到数据在哪里走偏，而不是只修报错那行
7. **最小修** — 只改修复根因所必需的部分
8. **验证** — 跑触发报错的用例确认修复，再跑相关测试确认无回归

## 临时插桩纪律

如果必须加临时 debug 日志：

1. 使用唯一前缀 tag，例如 `[DEBUG-a4f2]`。
2. 修复完成前用 `rg "\[DEBUG-a4f2\]"` 确认全部清理。
3. 不把临时日志、临时脚本或一次性输出提交进仓库。

## 合并冲突解决步骤

遇到 `git merge` / `rebase` 冲突时：

1. 列出冲突文件：`git diff --name-only --diff-filter=U`
2. 逐个读冲突文件，理解双方（`HEAD` 和 `incoming`）各自的意图
3. 解冲突时做**最小改动**，保留双方的合理意图，不要顺手重构
4. 解完后 `git add` 并验证项目能正常 build / test
5. 如果冲突涉及业务逻辑分歧，不要自决，列出战方意图问 507

## 完成后

修复验证通过后，不等于收尾。继续走 `/review` 审查改动质量与文档同步，再走 `/commit` 提交。
- 如果修复改了用户可见行为、版本语义或发布说明，提醒 507 同步 CHANGELOG / 版本号 / tag。
- 临时插桩和调试脚本必须清理干净后再进入 review。
