## 目标

目标不是"写 README"，而是让 agent 和人能快速理解项目结构、目录职责、入口、边界和导航。

项目地图有两个载体，都要管：

1. **`doc/` 目录组织** —— 按编号段划分职责，是项目知识库的骨架。
2. **分层 `README.md`** —— 每一层包含代码的目录都有一份，是代码层的导航。

这里关注**空间地图**（目录结构与代码层导航），不直接维护时间线文档。若校准地图时发现用户可见变化、版本语义或发布说明需要同步 `CHANGELOG.md`，在总结里标注出来，交给 `/review` 或 `/commit` 收尾。

## doc/ 目录组织（项目知识库骨架）

用户的成熟项目用一套编号目录约定（范本：`ai-database-vault/doc/README.md`、`pi-dteam/doc/README.md`）。本表是项目地图执行时的核心知识;权威标准定义见 `507-ground` 技能的 `references/project-standard.md`。

| 段 | 职责 | 权威性 |
|---|---|---|
| `00-产品与原则/` | 产品定位、目标用户、体验原则、UI 气质 | 长期有效权威 |
| `10-架构与运行/` | 当前架构、数据管道、运行、测试、开发规范 | 当前实现权威 |
| `20-能力参考/` | 外部能力事实库、实测数据、license、能力边界 | 事实参考,不决定排期 |
| `30-路线图/` | 版本维度编排、后续计划 | 路线图唯一权威 |
| `40-版本实施方案/` | 版本级落地方案,必须含 checklist | 受路线图约束 |
| `90-归档/` | 已完成或过时的历史方案 | 否,仅查历史事实 |

纪律:

- **版本实施方案必须内置 checklist**,不单独拆成另一份文档;前期拆分草稿开工前必须合并到权威方案。
- **每个版本优先只有一个权威实施方案**。
- **归档文档不作为当前权威**,冲突时以 `30-路线图/` 和 `10-架构与运行/` 为准。

处理原则:

- 小项目(< 5 个文档)可不强制分段，但必须在 `doc/README.md` 说明为何保持平铺。
- 中大型项目已有其他清晰组织方式,跟随现状,不强制重命名，但必须在 `doc/README.md` 标明当前实现权威、路线图权威和历史归档边界。
- 只要项目已有路线图、版本实施方案、历史归档或超过 5 个文档，默认提示采用这套编号段；不要只新建单个 `40-版本实施方案/` 而保留其他文档平铺。
- `doc/` 混乱(原则/路线图/方案混在一起)时,建议按这套编号段重构,先给用户看草案。

## 分层 README(代码层导航)

1. 如果指定了目录,只处理该目录及明确要求的子目录。
2. 如果没指定,先扫描当前工作目录,列出**包含代码文件但缺少地图载体(当前为 `README.md`)**的目录,等我确认后再动手。
3. 区分两种场景:
   - **补缺失**:目录有代码但没有 `README.md` → 写最小必要地图。
   - **校准过时**:已有 `README.md` 但和代码现状不符 → 只改失真部分,不重写整篇。

## 原则

1. **地图优先**:写目录职责、入口、主要文件、上下游关系、边界约束;不要写营销话术或泛泛介绍。
2. **遵循现状**:先读上下级和相邻 README,沿用既有风格。
3. **最小必要**:只写代码能支撑的事实;不为完整/好看扩写。
4. **诚实标注不确定**:读代码后仍判断不准就写"待确认"或省略,不编造。
5. **只改文档**:本任务不改代码;发现代码/文档冲突先报告。
6. **同步上下级**:本层职责变化时,检查上级目录表和下级引用是否也要同步。

## 流程

1. **读规范**:先读项目根 `AGENTS.md`(若有)里关于文档/README 的规范。
2. **读上下文**:读目标目录上下级 README、相邻 README。
3. **读代码**:识别入口、对外接口、主要文件、依赖关系、目录边界。
4. **先画地图**:按层级列出目录职责和缺口;范围太大时,先给候选目录,等我确认。
5. **写载体**:当前用 `README.md` 承载。常见骨架按需取舍:
   - 一句话定位
   - 职责 / 当前现状
   - 主要文件(文件 → 说明)
   - 入口 / 运行方式(仅当必要)
   - 边界 / 约束 / 上下游
6. **自检**:有没有写未读过的文件?含代码子目录是否仍缺地图?上级目录表是否需要更新?
7. **总结**:说明处理范围、补缺失还是校准、依据哪份相邻风格、哪些地方待确认。

## 红线

- 不自动扩大范围。
- 不硬套模板。
- 不写代码没支撑的章节。
- 不给单个文件单独建 README，除非该目录只有它一个核心文件且确实需要长说明。
- 不直接编辑 `CHANGELOG.md`；发现需要同步时，在总结里标注交给 `/review` 或 `/commit` 收尾。
