结合我们前面的讨论，以及以下补充内容，先帮我定位这个问题在项目里的位置，再解释相关上下文，最后进入讨论：

$@

这是项目内讨论的**起手式**。目标不是马上给答案、写方案或改代码，而是先让双方站到同一张地图上：这个问题属于哪个项目语境、牵涉哪些文件 / 概念 / 入口、真正需要讨论的点是什么。

## 先读上下文

避免凭记忆讲错。按需读用户给出的文件 / 目录 / 符号，以及项目根 `AGENTS.md`、`doc/README.md`、相关 `doc/术语表.md` / `doc/决策档案/`、目标代码目录及上下级 `README.md`、相关源代码 / 测试 / 配置。

没有明确路径时先做窄范围搜索定位候选文件，不要一上来全仓库漫扫。只能从局部判断时标注“基于当前已读文件”；不把未读过的实现细节说成事实。

## 判断用户真正要什么

区分这是：理解代码 / 概念、排查现象、设计讨论、实现前对齐，还是文档 / 项目地图讨论（README / doc / CHANGELOG / 术语表 / 决策档案）。

## 工作流路由

定位后给出推荐工作流。默认只定位和推荐，不直接改代码或写文件；只有 507 明确说“继续”“直接做”“按这个修”时，才按推荐模式进入执行。

- **普通新任务 / 需要持续做到完成**：走 `/dgoal`，让目标、阶段、任务和验证闭环接管执行；本 prompt 不新增普通执行入口。
- **bug / 报错 / 崩溃 / 合并冲突**：按 `/fix` 纪律继续，先建可重复反馈环，再最小根因修复。
- **测试任务 / 失败测试 / 覆盖率缺口**：按 `/test` 纪律继续；明确 test-first 时再走 `/tdd`。
- **局部复杂度 / 行为不变清理**：按 `/simplify` 纪律继续；暴露模块形状问题转 `507-inspection`。
- **README / doc / 目录职责 / 项目地图**：按 `/map` 纪律继续；全项目初始化或巡检转 `507-ground`。
- **决策没问透**：转 `507-grill`，边追问边把术语 / 决策沉淀进 `doc/术语表.md` / `doc/决策档案/`。
- **需要沉淀 PRD（需求规格）**：转 `507-blueprint`；**需要开 GitHub issue 工单**：转 `507-workorder`。
- **需要外部借鉴或参考调研**：转 `507-research`。

## 输出结构

按复杂度取舍，默认输出这些部分：

```md
## 一句话定位
<这个问题在项目里大概属于什么位置 / 层次。>

## 我已查看的依据
- `<path>` — <为什么相关>

## 项目地图
- 入口：...
- 相关模块 / 文档：...
- 上游 / 下游：...
- 关键术语：...

## 当前机制
<用项目语言解释现状：数据流、控制流、状态流、文档权威或运行方式。>

## 讨论焦点
- <真正需要讨论或确认的问题>

## 下一步
- 判断：<属于哪类问题>
- 推荐：<继续在当前对话展开，或转 507-grill、/fix、/test、/map、/dgoal、某个 skill>
- 原因：<为什么这条路最省决策成本>
- 先做：<推荐先聊 / 查 / 验证哪一件事>
```

用户明确只要“解释这段代码”时，可压缩为：一句话总结 → 调用地图 → 工作原理 → 边界情况；即使压缩也要先说明已读依据。

## 红线

- 不直接改代码，除非 507 明确要求进入实现。
- 不把猜测说成事实；没读过就标注不确定。
- 不跳过项目文档和相邻 README 直接解释源码。
- 不把本次输出写成最终方案；这里只负责定位、解释和打开讨论。
- 不问能通过读代码、文档或现有上下文自己回答的问题。
