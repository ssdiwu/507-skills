# 调研借鉴（research）

给定一个**外部参照物**，判断它对**当前项目**有什么借鉴意义，把判断沉淀成一份可维护的参考文档。

解决的核心问题是 **借鉴错位**——要么照抄别人、把当前项目改成了别人的样子；要么只看 README 就吹"这个很好"，没落到自己的场景；要么简单二分"借/不借"，漏掉"我们其实已经有了"和"思路对但不能整体搬"。

先读懂别人怎么做的，再带着自己的定位判断能借什么、借到哪、改哪里。

## 覆盖什么

- 给一个 GitHub repo / 文章 / 飞书 wiki / 本地项目 / 公众号 / 图片里的开源项目，问"对我们项目有没有借鉴意义/好处/启发"。
- 多个参照物对照（"前面是我做的，对比后面的"）。
- 借鉴结论 → 影响路线图 / 技术选型 / 架构方向的判断。
- 产出 `doc/20-能力参考/`（或项目约定目录）下的结构化参考文档。

## 不覆盖什么

- 解释用户不理解的单个概念或术语 → 用 `507-explain`
- 系统学习一个主题、建立学习计划或长期知识工作区 → 不在本技能范围
- 把对齐后的结论落成可执行规格 / 工单 → 不在本技能范围（`507-research` 只负责把外部机制搞清楚）
- 直接把别人代码复制进项目（本技能只产判断和改进点，复制实现走正常开发流程）
- 纯学术调研 / 市场竞品报告（本 skill 是工程师视角的"能不能用在我们项目"）

> 递进：`507-research`（看外部有什么可借）→ `507-grill`（把候选问透）→ 落成规格或工单（另一回事）。

## 核心纪律（不可违反）

1. **落点是"改我们"，不是"抄别人"**——借鉴判断必须落到"对我们项目的具体影响/改进点"。参照物只读不改；要改的是当前项目。不要去 patch 被调研的对象。
2. **带着当前项目定位做取舍**——先按 AGENTS.md 文档优先级读 `doc/` 和 `AGENTS.md`，搞清当前项目**是什么、不是什么**。借鉴不能违背项目定位（轻量化项目不借重量级方案，编排引擎不借流水线引擎）。
3. **不二分，分三档**——每个机制判断归入：**设计冲突**（明确不借）/ **同思路**（我们已有，印证方向）/ **候选**（值得吸收）。不要简单"借/不借"，也不要全盘肯定或全盘否定。
4. **候选必须落到具体改进点**——"值得吸收"的每一项都要能回答"改哪个文件 / 哪个模块 / 哪个决策"。停在"这个思路很好"不算完成。
5. **先读懂真实机制，再判断**——先下代码 / 抓文章 / 读设计文档，提炼"关键机制"（它怎么做、为什么这么做），不要只看 README 就开始评价。
6. **正反例 + 一手源优先**——单一来源容易偏信。调研必须：
   - **正反例**：同类机制至少查 3 个（佐证/延伸/不同角度），反例或替代方案至少 1 个（查不到也认，但默认要找）。
   - **一手源优先**：机制事实优先一手信息源（官方文档 / 源码 / 原始论文 / issue / 作者博客）。找到转载 / 二手解读没问题，但要继续深挖到它引用的一手源，**拿一手源做准**——转载源会把机制传歪，而 research 的结论直接影响路线图 / 技术选型。
   - **豁免**：用户明确指定"就看这个 repo / 这篇"时不强制正反例，但转载源即使指定也要追一手。

## 工作流程

### 1. 锚定三件事（能自己查的不问）

- **被调研对象**是什么——URL / 路径 / 文章 / 图片 / wiki。
- **借鉴目的**——影响路线图？某个功能的实现方式？架构 / 技术选型？还是单纯拓宽视野？一次只问一个，附推荐答案。
- **当前项目定位**——读 `doc/`、`doc/术语表.md`、`AGENTS.md`，不要问能自己查的。

### 2. 读懂被调研对象

- **GitHub repo**：clone 到临时目录或直接读，看目录结构、核心代码、README、设计文档；提炼它**怎么做**，不是它**有什么**。
- **文章 / 公众号 / wiki**：抓取正文，区分事实和观点。
- **图片**：OCR / 识别其中的项目名、架构图、代码片段。
- **本地项目**：读它的 `doc/` 和 `README.md` 优先于 grep 源码。

产出：一段"关键机制速览"——3~6 条它真正的做法和理由。

### 3. 联网核实 + 一手源深挖

默认要做（不是"按需"）——单篇 README 或一家之言下不出"值得借鉴"的结论。

**正反例**：用联网搜索查同类方案，至少 3 个同类 + 1 个反例 / 替代方案。对照确认被调研对象的做法是不是主流、有没有更好的。

**一手源深挖**：对每个关键机制事实，优先落到一手信息源（官方文档 / 源码 / 原始论文 / issue / 作者博客）。
- 搜到的是转载 / 二手解读 → 沿它引用的链继续深挖到一手源，拿一手做准。
- 一手源和二手解读矛盾时，**信一手**，并在参考文档里标注分歧。
-用户指定"就看这个"时，正反例可豁免；但转载源即使指定也要追到一手。

### 4. 能力映射 + 三档分级

针对当前项目的具体场景，逐个机制判断归入三档：

- **设计冲突（明确不借）**：和当前项目定位冲突，写清楚为什么不借（"我们轻量化，不引入流水线编排"）。
- **同思路（已有印证）**：当前项目已有类似机制，被调研对象印证了方向，不必再动。
- **候选（值得吸收）**：每条必须带"如果吸收，改哪里"。

### 5. 沉淀参考文档

写到项目约定的参考目录（常见 `doc/20-能力参考/`、`doc/research/`、`doc/参考/`）。目录不存在就惰性创建，命名和用户对齐一次。编号扫目录取最大 +1。骨架见附录。

### 6. 产物自检

写完参考文档后，从没有参与本次调研的项目维护者视角完整重读；发现问题直接修正文档，不把检查清单附在正文末尾：

- **事实可追溯**：每个关键机制事实都有可回到的一手来源；事实、来源观点与本项目推断明确分开。
- **对照充分**：同类机制、反例或替代方案达到本 skill 的要求；确实查不到时明确记录搜索范围和证据缺口，不用单一来源冒充共识。
- **三档完整**：设计冲突、同思路、候选三档都显式出现；总体判断与三档内容一致，不一边判定冲突一边列为吸收候选。
- **候选可落点**：每个候选都指出当前项目中受影响的文件、模块或决策；只有“这个思路很好”的内容降级为观察项。
- **范围与歧义**：没有 `TODO`、`TBD`、未替换占位符或把尚未核实的猜测写成结论；不把调研扩写成规格、工单或实施方案。
- **引用有效**：文档中的 URL、本地路径和代码位置能够打开，并指向实际支撑对应判断的内容。

关键机制仍无法核实时，降低结论置信度或明确停止判断；不要为了交付一份完整文档而补写确定性。

### 7. 提示下一步

高价值候选 → 建议走 `507-grill` 深挖，或落成规格 / 工单进入实施。不够格的候选留在文档里备查即可，不要硬推实施。

## 在用户系列里的位置

`507-research` 是**外部证据环节**——通常由 `507-explore` 识别出外部机制未知后进入，再把可靠事实交给对齐或规格化：

```
explore（识别外部未知）→ research（看外部可借鉴）→ grill（选哪个/怎么改）
                                                           ↓
                                                    规格 / 工单 / 备查
```

边界：

- **vs `507-explain`**：`507-research` 核实一个外部对象的真实机制并判断对当前项目的借鉴；`507-explain` 只把用户不理解的单个概念讲懂，默认不产参考文档。
- **vs `507-explore`**：`507-explore` 只读梳理项目内的已知、未知与当前 `frontier（可推进边界）`；发现需要系统核实外部对象时才路由 `507-research`。
- **vs 原型验证**：research 读懂别人；原型是自己造一个验证。
- **vs 架构体检**：research 找外部启发；架构体检找内部加深机会。research 的发现可作为架构体检的输入。

## 启动姿势

当用户给出一个外部参照物并问"对我们有没有借鉴意义 / 好处 / 启发"时：

1. 确认借鉴目的（一次一问，附推荐答案）；同时读 `doc/` 和 `AGENTS.md` 掌握当前项目定位。
2. 读懂被调研对象，提炼"关键机制速览"。
3. 联网核实 + 一手源深挖（同类 3+ / 反例 1+，机制事实追一手）。
4. 做三档分级判断，候选落改进点。
5. 沉淀参考文档到项目约定目录。
6. 完成产物自检并直接修正事实归属、对照覆盖、三档一致性、候选落点、歧义和失效引用。
7. 给一句话总体判断 + 下一步建议（对齐深挖 / 落规格 / 备查）。

## 附录：参考文档骨架

来自 `pi-dteam/doc/20-能力参考/25-swarm-ide蜂群IM编排参考.md` 的成熟形态：

```md
# N-{对象名}参考

## 1. 总体判断
一句话：值得 / 不值得 / 部分值得 + 为什么。

## 2. 核心机制速览
提炼对象的关键机制（3~6 条），写"怎么做 + 为什么"，不罗列功能。

## 3. 与 {本项目} 的关系

### 3.1 设计冲突（明确不借）
每条带项目定位理由。

### 3.2 同思路（已有，印证方向）
当前项目已有的类似机制，印证了方向。

### 3.3 候选（值得吸收）
每条带"如果吸收，改哪个文件 / 模块 / 决策"。

## 4. 可借鉴的具体文件 / 代码 / 资源
指向对象里值得精读的具体位置。

## 5. 决策记录
够格进 `doc/决策档案/` 的在此留索引（文件名 + 一句话），正文写进 ADR。
```

规则：

- 总体判断放最前，先给结论。
- 三档判断是文档主体，缺任何一档都要显式说明（"未发现设计冲突"也要写），不要默默跳过。
- 候选不落改进点不算候选，降级为"观察项"。
- 决策记录只留索引，够格的（难逆转 / 无上下文会困惑 / 有真实权衡）才进 `doc/决策档案/`。
