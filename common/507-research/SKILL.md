# 外部调研（research）

把一个外部问题或参照物研究清楚，产出可被下一步直接消费的可靠证据。

`507-research` 是 `common/`（横向层）的外部证据入口，不只服务代码，也服务概念解释、项目探索、写作挖掘与方案决策。它回答的是：

> 外部世界目前能确认什么，证据在哪里，还有什么不能确认？

## 输入

- 一个待核实的问题、对象或主张；
- 证据将服务的当前目标，例如理解概念、评估项目、形成碎片或支持决策；
- 用户限定的来源、时间或范围（若有）。

没有明确当前项目时也能独立调研；需要判断“对本项目能借什么”时，再读取项目 `AGENTS.md`、`doc/` 与相关权威说明。

## 覆盖什么

- 外部项目、源码、官方文档、文章、论文、标准、竞品和参考实现；
- 时效性事实、机制核实、多个方案对照与反例；
- 对当前项目的借鉴判断；
- 为后续 skill 提供结构化的 `evidence handoff（证据交接）`。

## 不覆盖什么

- 把材料变成写作碎片或观点 → `507-mine`；
- 把用户不理解的概念讲懂 → `507-explain`；
- 探索项目内部已知、未知和当前可推进边界 → `507-explore`；
- 替用户决定产品意图或风险取舍 → `507-grill`；
- 直接写规格、工单、正文或实现。

## 核心纪律

1. **先定义证据问题**：明确要核实的主张及证据将支持什么，不为“资料更多”无限扩搜。
2. **一手来源优先**：机制事实优先官方文档、源码、原始论文、标准、原始数据、作者说明或权威记录；二手来源用于发现线索和补充语境。
3. **分开事实与判断**：明确区分已核实事实、来源观点、本项目推断和仍未知事项。
4. **深度与风险相称**：简单事实用足够的一手证据即可；会影响路线、架构、产品或公开结论时，再扩大到多个同类样本、替代方案和反例。不要机械要求每次都“3+1”。
5. **冲突证据不抹平**：来源相互矛盾时写明分歧、时间与适用范围，降低置信度；不能确认就保留未知。
6. **默认轻产物**：通常在对话中交付证据包。只有用户要求、项目规则规定，或结论需要长期复用时，才写入项目既有参考目录；不为一次调研新建文档体系。

## 工作流程

1. 锁定证据问题、使用场景和范围；能从上下文确定时不反问。
2. 先读指定对象及其一手来源；技术问题优先官方文档与源码。
3. 按结论风险补充同类样本、替代解释、反例或时效核验。
4. 交叉核对来源，标记时间、适用范围、冲突和证据缺口。
5. 输出证据交接，并根据当前目标选择下一步。

## 证据交接

按复杂度裁剪，最小结构是：

```md
## 结论
<当前证据支持的最短结论>

## 已核实事实
- <事实> — <一手来源>

## 推断与适用边界
- <从证据得到的推断，以及它只在什么条件下成立>

## 反例 / 替代解释
- <确有必要时列出>

## 未知与置信限制
- <仍无法确认的部分>

## 对当前目标的意义
- <下一步可直接消费的证据或约束>
```

若任务是“对当前项目有什么借鉴”，再把机制分成：设计冲突、同思路、候选；候选必须指出受影响的项目能力或决策，不停在“这个思路很好”。

## 完成与接力

- **完成信号**：目标主张已有可追溯证据，事实、推断与未知已分开，证据深度与决策风险相称。
- **产物**：对话中的证据交接；必要时是项目既有参考目录中的耐久文档。
- **候选出口**：需要讲懂时进入 `507-explain`；需要重新计算项目未知时返回 `507-explore`；需要形成写作碎片时进入 `507-mine`；需要用户取舍时进入 `507-grill`；前提已经充分时进入 `507-prd`、`507-issue`、写作或实施；只需答案时直接结束。
- **回退条件**：关键事实仍无法核实时，保留未知或降低置信度，不把猜测交给下游当结论。

## 红线

- 不把搜索结果摘要冒充事实核验；
- 不用二手解读覆盖一手来源；
- 不把当前项目推断写成外部对象的客观事实；
- 不为了形式完整强制落文档或堆来源；
- 不挖碎片、不替用户做决策、不顺手实施。
