# common/ 通用 skill

场景无关的 skill，代码工作流和写作工作流都用。

## 为什么有这一层

有些动作不分场景——代码项目和写作 vault 都要对齐、都要调研外部对象、都会遇到不懂的概念。这些 skill 不属于 write 流也不属于 code 流，放 common/，在两条流里扮演不同角色。

## 每个 skill 的介绍

#### `507-grill` 决策树追问
开工前对齐。把一个改动/方案的每个决策分支逐个问到底，边问边把术语和决策沉淀进 doc/。
- 在**代码流**里：对齐代码改动，沉淀进项目的 `doc/术语表.md`、`doc/决策档案/`、`doc/经验笔记.md`
- 在**写作流**里：拆解外部文章/对齐选题判断（grill 不直接产碎片，但能把判断问透，问透后另行落碎片）
- 核心纪律：一次只推进一个问题；需要用户决策时给 2–5 个有真实权衡的候选、标出推荐项并走遍决策树

#### `507-research` 调研借鉴
调研一个外部参照物（GitHub 项目、文章、飞书文档、本地项目、公众号、图片里的开源项目），判断它对当前项目有什么借鉴意义，产出结构化参考文档。
- **正反例 + 一手源优先**：同类机制至少查 3 个、反例 / 替代方案至少 1 个；机制事实优先一手源（官方文档 / 源码 / 原始论文 / issue / 作者博客），找到转载 / 二手解读要继续深挖到它引用的一手源拿一手做准
- 在**代码流**里：调研别人的库/项目能不能用在我们项目
- 在**写作流**里：调研外部文章/项目的真实机制（不深挖机制，深挖归 research）
- 产物是参考文档（不是碎片，不是选题）

#### `507-teach` 概念解释器
用一个不懂的概念/术语/技术名词，用各种通俗易懂的话、比喻、类比、白话讲到你懂为止。
- 场景无关：代码术语、写作概念、聊天疑问都用得上
- 默认不落盘（懂了就完）；值得长期记的可选落术语表
- 不挖碎片、不系统沉淀主题

## 和两条流的关系

- **grill** 是两条流的"对齐入口"——任何改动/方案/判断开工前都可以 grill
- **research** 是两条流的"深挖工具"——需要核实外部对象真实机制时调用
- **teach** 是两条流的"理解工具"——遇到不懂的概念随时调用

这三个 skill 不抢 write/code 流内部 skill 的活，是横向支撑层。
