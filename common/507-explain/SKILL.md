# 概念解释（`explain`）

把一个用户不理解的概念讲到真正理解。目标不是罗列完整知识，而是用最短定义和最合适的示意建立正确心智模型。

`507-explain` 做的是：**不懂的概念 → 清晰、可复述的理解**。

## 覆盖什么

- 技术概念、方法论、行业术语、缩写与框架名词；
- 两个或少量相近概念的差异与边界；
- 需要通过流程、状态变化、层级、映射或组件关系才能讲清的概念；
- 代码、写作或日常对话中出现的通用理解障碍。

## 不覆盖什么

- 定位项目里的问题、代码机制、证据与未知 → 用 `507-explore`；
- 系统调研一个外部项目、文章或参考实现 → 用 `507-research`；
- 对齐一个方案中的用户决策 → 用 `507-grill`；
- 系统学习整个主题、建立学习计划或长期知识工作区 → 不在本技能范围。

## 核心纪律

1. **讲懂，不讲全**：成功标准是用户能够形成正确心智模型或用自己的话复述核心，不是覆盖概念的全部历史、分支和进阶用法。
2. **先短后深**：先用一句白话定义建立锚点；仍有理解缺口时，再逐步增加比喻、场景、反例、相近概念对照或技术细节。
3. **从用户已知出发**：能从上下文判断背景时直接适配；背景差异会实质改变解释方式时，才一次询问一个简短问题。
4. **一次一个承重概念**：相关概念只在解释当前核心所必需时引入，不把回答扩成术语瀑布。
5. **示意服务理解**：存在三项以上映射、依赖步骤、状态变化、分支、层级或多方交互时，优先使用一个最小可用视觉；简单事实或一步关系不机械画图。
6. **区分事实与类比**：明确哪些是概念本身，哪些只是帮助理解的比喻；不能让类比反过来制造错误规则。
7. **默认不落盘**：解释完成即结束。只有用户明确要求，或该词已经成为项目稳定术语时，才按项目规则写入术语表。

## 选择示意方式

只选择最能降低理解成本的一种，不同时堆叠多种重复图：

| 关系形状 | 首选形式 |
|---|---|
| 精确字段映射、少量概念对比 | 表格 |
| 三步以上流程、状态变化、分支或多方交互 | `Mermaid（图表语法）` 流程图、状态图或时序图 |
| 小型层级、终端环境或纯文本更清楚的结构 | `ASCII（字符图）` |
| 单一事实、一步动作、一个定义 | 简短文字说明，不画图 |

图后用一两句指出读图重点；不把图中所有文字再完整复述一遍。当前界面不能可靠渲染 `Mermaid（图表语法）` 时，使用等价的 `ASCII（字符图）` 示意。

## 工作流程

1. **识别承重概念**：确认用户真正卡住的是哪个词、关系或差异；能从上下文确定时不反问。项目特有术语先读项目术语表与相关权威定义，通用概念不为形式完整漫扫项目。
2. **给最短定义**：一句话说明“它是什么”，避免用更多未解释术语定义它。
3. **选择解释路径**：根据概念形状与用户背景，在场景、比喻、反例、对比或最小视觉中选择最有帮助的一种。
4. **压测边界**：说明一个最容易混淆的“不是什么”或相近概念差异，防止只记住表面比喻。
5. **按反馈迭代**：用户仍未理解时换一种解释方式，不重复同一套措辞；理解成立就停止。

## 输出形态

简单概念可以只有两三段。复杂概念按需使用：

```md
## 一句话
<最短白话定义>

## 示意
<一个表格、流程图或字符图；确有必要时才用>

## 放进场景
<一个具体例子>

## 容易混淆
<它不是什么，或和相近概念的关键差异>
```

## 与相邻技能的边界

| 技能 | 分工 |
|---|---|
| `507-explain` | 用户不理解一个概念 → 讲懂 |
| `507-explore` | 项目问题尚未看清 → 读取证据，确定认知边界与下一步 |
| `507-research` | 外部参照物 → 核实机制并判断借鉴意义 |
| `507-grill` | 已知前提下的用户决策 → 追问并收束 |

## 红线

- 不用更多未定义术语解释当前术语；
- 不为展示完整而扩成系统课程；
- 不把类比当成事实边界；
- 不用无助于理解的装饰性图表；
- 不替项目探索、外部调研或决策对齐执行。
