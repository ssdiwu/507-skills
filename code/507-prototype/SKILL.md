# 原型验证（prototype）

Prototype（原型）是**回答一个问题的可丢弃代码**。问题决定形状：

- **逻辑 / 状态问题** → 做一个可交互 terminal app（终端应用），让用户按按钮推状态。
- **UI / 视觉问题** → 在一个路由上做 3 个结构差异明显的变体，用 `?variant=` 切换。

原型不是半成品。它的结论值得保留，壳应该删除或吸收。

## 覆盖什么

- 验证 state machine（状态机）、数据模型、API 形状是否顺手
- 在正式实现前试几个 UI 结构方向
- 让用户亲自"玩"一下状态或界面，发现纸面设计看不出的不适感

## 不覆盖什么

- 实现前把问题问透 → 用 `507-grill`（prototype 是对齐的下游：先问清要验证什么，再试做回答）
- UI 视觉质量深度评审 → 用 `impeccable`
- 正式生产实现 → 用普通开发流程
- 把原型结论拆成工单 → 用 `507-issue`

## 通用规则

1. **先写清楚问题**：原型顶部或旁边 `NOTES.md` 写一句"这个原型要回答什么"。
2. **从第一天标记为 throwaway**：产出放在 `原型/` 文件夹下（路径 / 文件名包含 `原型`），让读者一眼知道不是生产代码。
3. **一个命令能跑**：优先接入项目已有 task runner（`pnpm` / `npm` / `bun` / `python` / `make` / `just`）。没有就把命令写在原型顶部或 `NOTES.md`。
4. **默认不持久化**：state（状态）放内存。除非问题就是持久化，否则不接真实 DB；必要时用 `PROTOTYPE_WIPE_ME.*` 之类明确可删的本地文件。
5. **跳过生产级打磨**：不写测试、不做抽象、不做完整错误处理。目标是快、能回答问题、然后删。
6. **每一步暴露完整状态**：逻辑原型每次动作后打印状态；UI 原型每次切换后能看见完整页面。
7. **回答完就删除或吸收**：保留结论，删除壳；若暂时不能删，留下 `NOTES.md` 写清 verdict（结论）和下一步。

## 分支一：逻辑原型

适用问题：

- "这个状态机覆盖 X 后再 Y 的情况吗？"
- "这个数据模型能表达这个边界吗？"
- "这个 API 用起来顺吗？"
- "我想按按钮看看状态怎么变。"

流程：

1. **写明问题**：一段话说明要验证的 state model（状态模型）和核心疑问。
2. **选项目已有语言/运行时**：不要为了原型引入新包管理器或 runtime。
3. **把逻辑隔离成可移植纯模块**：reducer、state machine、纯函数集合，或小 class。它可以之后被吸收进正式代码；TUI 壳是可删的。
4. **做最小 TUI**：一屏上半显示当前 state，下半显示快捷键。每次按键后清屏重渲染全状态。
5. **交给用户试玩**：给出 run command；用户说"等等，这不该发生"的时候，就是原型抓到了想法里的 bug。
6. **记录答案**：结论写进 `NOTES.md` / issue / ADR / commit message，然后删壳或吸收逻辑。

逻辑形状优先级：

- 离散事件 + 单状态 → pure reducer：`(state, action) => state`
- 动作合法性本身是问题 → 显式 state machine
- 无隐式当前状态 → 一组纯函数
- 真正需要长期内部状态 → 小 class / module，接口保持清楚

反模式：

- 不要把 reducer / state machine 和 `console.log`、prompt、ANSI 控制码混在一起。
- 不要接真实数据库。
- 不要为原型写测试。
- 不要泛化未来需求。

## 分支二：UI 原型

适用问题：

- "这个页面应该怎么排？"
- "试几个 dashboard 方向。"
- "设置页要不要侧栏？"
- "我想先看几个完全不同的设计。"

流程：

1. **默认做 3 个变体**：超过 5 个会变噪音。变体必须结构不同，不只是颜色或文案不同。
2. **优先挂在现有页面**：已有 `/settings` 就在同一路由用 `?variant=A/B/C` 切换，保留真实 header、sidebar、数据、权限、密度。
3. **最后才新建 throwaway route**：只有确实没有宿主页面时，才建 `/原型/<name>` 或项目约定下的等价路径。
4. **写 Variant 组件**：例如 `VariantA` / `VariantB` / `VariantC`。每个变体有不同信息架构、布局或主交互。
5. **加浮动 switcher**：底部居中，左右切换，URL search param 同步，可分享、刷新稳定。
6. **生产环境隐藏 switcher**：必须 gate 在 `NODE_ENV !== 'production'` 或项目等价机制下。
7. **选中后清理**：删除输家和 switcher，把胜出方案重写进正式页面。

UI 反模式：

- 三个变体只是换颜色 / 卡片微调。
- 多个变体共享同一个 layout，导致根本没有结构差异。
- 原型接真实 mutation（写操作）。UI 原型默认 read-only，需要 mutation 就打 stub。
- 把原型代码直接升生产。应吸收决策，不吸收粗糙实现。

## 输出格式

做完原型后必须给用户：

```md
## 原型问题
<这段原型要回答什么>

## 运行方式
`<one command>` 或 URL

## 如何试
- 操作 1
- 操作 2
- 看哪个状态 / 页面

## 需要用户反馈的问题
- 哪个状态/变体符合预期？
- 哪个地方感觉不对？
- 是否删除、吸收或继续改原型？
```
