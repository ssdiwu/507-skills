# GitHub Issue（issue）

把任务结构化成**可上传 GitHub、可被 AI 领取执行的工单**。

解决的核心问题是 **任务留档与协同**——当你需要把一个任务交给 GitHub 上的 AI 或协作者领取时，口头描述不够，要落成一份结构化、自足、可追踪的 issue。issue 既是留档载体，也是执行合同：AI 领取后照着 issue 内容直接做，不需要你在 issue 之外再补一份执行说明。

`507-issue` 做的是：**任务 → 结构化 GitHub issue 工单**。

## 覆盖什么

- **正向**：把方案 / PRD / 对话拆成可领取的 issue（垂直切片，窄但完整的端到端路径）
- **反向**：把 bug / gap / 坏行为落成可追踪的 issue
- 多个 issue 的推进顺序与依赖总表

## 不覆盖什么

- 实施前把方案问透 → 不在本 skill 范围（先问透再创建 issue）
- 把对话/方案落成**需求规格文档（PRD）** → 用 `507-prd`（PRD 是"想清楚要做什么"，工单是"把要做的事写成可领取的单元"）
- 真实调用 issue tracker 创建 issue（除非项目已有 `gh` CLI 等明确工具与权限）
- 执行本身（领取后怎么做走执行入口，不在本 skill 范围）

> 和 `507-prd` 的关系：`507-prd` 产 PRD（需求规格），`507-issue` 产可领取 issue。PRD 可作为 `507-issue` 的可选输入（重任务先 PRD 再拆 issue），但 `507-issue` 也可独立触发（bug 可直接创建 issue，不经 PRD）。

## 核心纪律（不可违反）

1. **issue 即执行合同**——issue 要写得足够完整，让领取的 AI 照着内容直接做，不依赖你在 issue 之外再补说明。含：当前状态、目标行为、范围、验收、依赖。这就是 Agent Brief 该做的事——**融进 issue，不单独产**。
2. **垂直切片**——正向 issue 每个是一条窄但完整的端到端路径，能独立 demo 或验证。避免按 schema / API / UI / 测试等水平层拆。
3. **耐久优先于精确**——issue 可能几天后才被领取，代码结构会变。写接口、类型、行为契约、验收标准；少写文件路径，更不写行号。
4. **写行为，不写步骤**——描述系统应做什么，不规定领取者去哪个文件怎么改。
5. **用项目语言**——先读 `doc/术语表.md` 和相关 `doc/决策档案/`；issue 标题和正文使用项目术语。
6. **不空访谈**——综合已有上下文（对话/PRD/代码），不重复询问用户。缺关键事实时，一次只问一个问题，并附推荐答案。
7. **范围外要显式**——明确写"不做什么"，防止领取者镀金（擅自加东西）。
8. **远程元数据默认必填**——上传 GitHub 后，每个 issue 都必须按仓库现有规范设置并核验合适的 `labels（标签）` 与 `assignees（负责人 / 拥有者）`。先读仓库规则、issue 模板、现有标签和同类 issue；已有负责人约定时按约定执行，没有约定时默认当前 GitHub 登录用户，用户明确指定时以指定为准。只有当前账号或 `token（访问令牌）` 缺少对应仓库权限时才允许元数据未落地；此时必须报告权限错误、待补标签与负责人，并把远程元数据标为未完成，不得声称 issue 已收口。

## 模式一：正向 issue（把方案/PRD 拆成可领取工单）

适用：用户说"拆 issue""开 issue""把方案传 GitHub""让 AI 领取"。

### 正向 issue 八板块模板

来自 经实践验证的 issue 模板（如同类项目的既有 issue），这是验证过的成熟形态：

```md
## 1. 标识 [identity]
- **id**: 唯一标识（若项目有 id 规则则遵循，否则用 issue 编号）
- **title**: 人类可读名称
- **type**: 对象类别（infra / feature / bug / gap / 治理 等）

## 2. 背景与目标 [background]
- **why**: 为什么要做，承接哪个上层目标/issue
- **problem**: 不做会怎样
- **goal**: 这一轮要得到什么（一句话）

## 3. 当前状态快照 [current_state]
- 当前现状（已有什么、还缺什么）
- 已拍板补充约束（若有）

## 4. 范围与不做 [scope]
### includes
- 这一轮要做的
### excludes
- 明确不做的（防止镀金）

## 5. 关系与依赖 [relations]
- **depends_on**: #...（前置 issue）
- **blocks**: 后续什么依赖这个
- **related_issues**: #...

## 6. 验收与验证 [acceptance]
### criteria
- [ ] 标准 1（可独立验证）
- [ ] 标准 2
### verification
- 怎么验证（由谁/怎么核对）

## 7. 执行与决策 [execution]
- **decision**: 已定方向（若有）
- **next_action**: 建议先做什么

## 8. 远程映射 [remote]（若上传 GitHub）
- **parent_issue**: #...
- **role**: 这个 issue 在整体里的角色
- **labels**: 遵循仓库 `AGENTS.md`、贡献规范、issue 模板、现有标签词表与同类 issue 的用法；从仓库实际采用的任务类型、优先级或领域维度中选择能准确表达任务的最小标签集合，不照搬其他仓库的分类，也不自行创建新标签。
- **assignees**: 遵循仓库已有 ownership（归属）或分派约定；没有明确约定时默认当前 GitHub 登录用户，用户明确指定时以指定为准。
- **metadata_status（元数据状态）**: `complete（完成）`；若仅因仓库权限无法落地，写 `permission_blocked（权限阻塞）`，并列出待补标签、待补负责人和真实权限错误。
```

### 可选：推进顺序与依赖总表

当一次拆出多个 issue 时，在其中一个主 issue（或单独文档）里附推进顺序：

```md
## 推进顺序与依赖总表
### 推荐线性推进顺序
1. #A — ...
2. #B — ...
3. #C — ...

### 关键依赖关系
- #A 为 #B/#C 提供前置锚点
- #B 先于 #D，因为 ...

### issue 类型分组
- 开路 issue: #A, #B
- 内容层 issue: #C, #D
- 最终 issue: #E
```

## 模式二：反向 issue（bug/gap 转工单）

适用：用户说"记个 bug""这个行为不对帮我转 issue""这个 gap 要落成 issue""提个 QA"。

### 反向 issue 模板

来自 经实践验证的 gap issue 模板（如同类项目的既有 gap issue）：

```md
## 现象
客观描述实际发生了什么（不掺判断）。
- 具体行为
- 数据/日志证据

## 为什么这是个问题
| 维度 | 影响 |
|------|------|
| 功能 | ... |
| 性能 | ... |
| 可读性 | ... |
| 正确性风险 | ... |

核心矛盾：设计定位说 X，实现上却是 Y。

## 前置依赖
无 / #...（这是独立 gap / 依赖某 issue）

## 方向（待决策）
### 方案 A：...（推荐）
- 优点 / 缺点 / 改动范围

### 方案 B：...
- 优点 / 缺点 / 改动范围

## 验收标准（方案确定后补充）
- [ ] ...
- [ ] ...

## 风险提示
- 需确认的隐式假设
- 可能受影响的相邻功能
```

### 反向 issue 纪律

- **轻澄清，不过访**：最多 2-3 个短问题，补齐三件事——实际发生了什么、预期是什么、怎么复现。
- **用户视角**：写行为，不写代码实现；不用文件路径、行号或内部模块名（除非是定位必需）。
- **复现步骤必填**：还原路径不清楚时，先补步骤再写 issue。
- **先判单 issue 还是拆分**：一个坏行为 → 单 issue；多个独立坏点 → 拆成多个薄 issue。

## 工单自检

发布或交付本地草稿前，切换到领取者视角完整重读；发现问题直接修正文，不把检查清单贴进 issue：

- **自足**：不读取原聊天也能知道当前状态、目标行为、范围、不做、依赖和验收。
- **无占位**：没有 `TODO`、`TBD`、空字段、伪造编号或尚未创建却写成真实链接的依赖；需后补的远程编号有明确发布顺序。
- **垂直切片**：一个正向 issue 只承载一条可独立演示或验证的端到端结果；多个独立结果已拆分。
- **行为合同**：正文描述做什么和如何验收，不把文件路径、行号或实现步骤当作合同。
- **前后一致**：标题、目标、includes、excludes、依赖、验收、标签和负责人没有互相冲突；反向 issue 的现象、预期与复现步骤能对应。
- **可复验**：每条验收标准都能由领取者或审查者独立判断通过或失败，不使用“完善”“优化”“支持”等无判定条件的表述。

若仍缺会改变产品意图、范围或风险承受的关键决定，停止发布并回到 `507-grill`；不要把未对齐任务包装成可领取工单。

## 发布

- 按依赖顺序发布：blocker 先发，后续 issue 才能引用真实编号。
- 发布前先读仓库 `AGENTS.md`、贡献规范与 issue 模板，再用 `gh label list` 读取已有标签；必要时查看少量同类 issue，确认标签和负责人惯例，不把其他仓库的词表套进来，也不自行创建新标签。
- 找不到任何合适的现有标签时，在创建 issue 前停止并说明标签体系缺口；只有仓库规则已明确给出待建标签或用户另行授权时才创建标签，不先发布无标签 issue。
- 用 `gh api user` 确认当前登录用户；仓库有 ownership（归属）或分派约定时按约定选负责人，否则使用当前登录用户。
- 用 `gh issue create` 等工具上传（若项目有 gh CLI 和权限）。
- 创建成功后必须补齐远程元数据：`gh issue edit <number> --add-label "..." --add-assignee "..."`。
- 创建后必须用 `gh issue view <number> --json labels,assignees` 核验标签和负责人确实已落远程。
- 若 issue 已创建、但标签或负责人因仓库权限设置失败，保留已创建 issue，报告 `URL（链接）`、真实权限错误和待补元数据，把状态标为 `permission_blocked（权限阻塞）`；不得静默跳过、伪造完成或把权限错误扩大成自行创建标签、修改仓库权限。
- **不关闭或修改 parent issue**。

## 和其他 skill 的边界

| skill | 分工 |
|---|---|
| **issue** | 任务 → GitHub issue（本 skill） |
| **prd** | 对话/方案 → PRD 需求规格（`507-issue` 的可选上游；PRD 想清楚，再拆 issue） |
| **grill** | 把方案/改动问透（创建 issue 前对齐） |
| **执行入口** | 编排执行（领取 issue 后另行编排；本 skill 只产 issue，不编排执行） |
| **修 bug** | 修复走执行入口（反向 issue 记录问题后，修复另行处理） |

时序：对齐 → 写 PRD（可选）→ 创建 GitHub issue → AI 领取 → 执行。

## 启动姿势

当用户：
- 说"拆 issue""开 issue""把方案传 GitHub""让 AI 领取"→ 正向模式
- 说"记个 bug""这个 gap 落成 issue""提个 QA"→ 反向模式

你就：
1. 识别正向还是反向。
2. 读 `doc/术语表.md` 和相关上下文，用项目语言。
3. 按对应模板写 issue（issue 即执行合同，要自足完整）。
4. 多个 issue 时附推进顺序与依赖总表。
5. 完成工单自检并直接修正占位符、切片、行为合同、一致性和不可复验表述。
6. 按依赖顺序发布（有工具就上传，没有就本地文档），并核验每个远程 issue 的仓库适配标签与负责人；仅权限不足时显式报告 `permission_blocked（权限阻塞）`。

## 红线

- 不产 Agent Brief 作为独立文档（融进 issue，issue 本身就是执行合同）。
- 不写文件路径/行号作为执行指令（写行为契约，耐久优先）。
- 不把 issue 写成实现步骤（写"做什么"不写"怎么改"）。
- 不跳过范围外声明（excludes 防镀金）。
- 不在和 `507-prd` 的边界上抢 PRD 的工作。
- 不用固定标签词表覆盖仓库现有规范，也不未经授权创建新标签。
- 不在标签或负责人未经远程核验时声称 issue 已收口；权限不足是唯一例外，且必须显式报告未完成元数据。
