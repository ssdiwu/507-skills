# code/ 代码工作流

代码工作流按建造隐喻组织：勘察现场、夯实地基、画蓝图、开工、检查、改造、验收和收尾。每个 skill（技能）都可独立触发；完整链路是导航，不是强制仪式。

## 工作流地图

```text
507-survey 勘察项目语境
        │
        ├─ 507-ground 地基初始化/全量巡检（环境维度，可独立运行）
        ├─ 507-grill 对齐真实决策（common）
        ▼
507-blueprint 需求蓝图（可选）
        ▼
507-workorder 可领取工单（可选）
        ▼
507-mockup 可丢弃样板（按需）
        ▼
正常实施，或选择专门施工动作：
  507-repair    已知故障的根因修复
  507-probe     测试资产与失败诊断
  507-formwork  明确 test-first 时的红绿重构
  507-retrofit  外部行为不变的内部与接缝改造
        │
        ├─ 507-asbuilt 校准实况地图文档（按需）
        ▼
507-audit 审查明确交付范围
        ▼
507-closeout Git 收尾（仅在明确要求提交时）

周期性架构回路：
507-inspection 只读体检 → 完整证据报告 → 507-retrofit 逐项验证、施工或关闭 → 507-audit
```

`507-handover`（会话移交）位于 [`common/`](../common/README.md)，任何阶段需要中断、换会话、压缩上下文或转交时都可使用。

## 环境与入口

### `507-survey` 勘察

项目内讨论的只读起手式。先读文档和代码依据，定位入口、上下游与当前机制，再推荐后续路径；默认不实施。

触发：解释项目、定位代码、先看懂、先讨论。

### `507-ground` 地基

新项目初始化或全量项目规范巡检。核对 `AGENTS.md`、入口 README、CHANGELOG、doc、术语、决策、代码目录和验证现实；先报告再路由，不修改业务代码。

针对某个目录的 README/doc 地图维护不跑 ground，使用 `507-asbuilt`。

## 设计与任务化

### `507-blueprint` 蓝图

把对话或方案沉淀成 PRD（产品需求文档），回答解决什么问题、给谁、怎么验收。轻任务可跳过。

### `507-workorder` 工单

把需求、bug 或 gap（缺口）整理成可上传 GitHub、可被 agent 领取的 issue（问题单）。issue 本身就是执行合同。

### `507-mockup` 样板

用 throwaway prototype（可丢弃原型）回答一个具体设计问题。逻辑/状态模型走可交互终端原型，UI（界面）方向走可切换视觉变体；得到答案后删除壳或吸收结论。

## 专门施工动作

### `507-repair` 修复

用于已明确的 bug、报错、崩溃、行为回归或合并冲突。先建立可重复反馈环，再做最小根因修复并运行相关回归。

### `507-probe` 探测

测试本身是主任务时使用：补行为测试、运行测试、修正过时测试、分类和缩小失败范围。真实生产 bug 转 `507-repair`。

### `507-formwork` 模板施工

仅在用户明确要求 TDD（测试驱动开发）、test-first（测试先行）、先写测试或红绿重构时使用。按垂直切片逐个完成 RED → GREEN → REFACTOR。

### `507-retrofit` 既有结构改造

在公开 API（接口）、契约和调用者可观察行为不变的前提下，深改内部模块、抽象与接缝。可直接处理指定范围，也接收 `507-inspection` 的完整报告；每项先建基线和验证信号，成立才施工，不成立带证据关闭。

需要改变产品行为或公开契约时退出 retrofit，回到需求与正常实施流程。

## 文档、验收与收尾

### `507-asbuilt` 实况地图

以当前代码为证据，创建或校准根/分层 README、`doc/README.md` 和地图结构。它读取代码但只修改地图文档，不接管业务代码、术语、决策、路线图或变更日志。

### `507-audit` 审查

对任意明确变更范围统一检查 Standards（项目规范）、Spec（需求符合度）和 Code Quality（代码质量）。工作区干净时仍可审查分支、提交或路径；只报告发现和路由，不直接修。

### `507-closeout` 收尾

用户明确要求 commit（提交）时，轻量复核差异、同步必要产物、运行验证、精确暂存并创建真实本地 Conventional Commit（约定式提交）。默认不 push（推送），不重复完整 audit。

## 架构体检回路

### `507-inspection` 巡检

只读寻找浅模块、职责泄漏、假设接缝、难测接口和局部性问题。每张候选卡必须提供代码观察证据、置信等级、公开行为不变量、设计约束和验证缺口。

inspection 不施工；完整报告交给 `507-retrofit`。retrofit 为每项建立第二层证据门，结论可以是“施工并通过”“带证据关闭”或“需要行为变化，转需求实施”。

## 通用支撑

- `507-grill`：只有产品意图、权责边界、不可逆成本或风险取舍需要用户决定时，进行决策树对齐。
- `507-research`：研究外部项目、文章或参考实现的真实机制和借鉴意义。
- `507-teach`：把单个陌生概念讲懂，不替代项目勘察。
- `507-handover`：中断、换会话、压缩上下文或转交时在对话中输出临时状态，不写文件。

## 路由判断

| 任务信号 | 使用 |
| --- | --- |
| 先定位、解释、讨论项目问题 | `507-survey` |
| 初始化或全量规范巡检 | `507-ground` |
| README/doc/目录地图失真 | `507-asbuilt` |
| 已知 bug、报错或冲突 | `507-repair` |
| 测试是主任务 | `507-probe` |
| 明确要求 test-first/TDD | `507-formwork` |
| 行为不变的内部简化或架构改造 | `507-retrofit` |
| 主动寻找架构加深机会 | `507-inspection` |
| 审查某段明确交付 | `507-audit` |
| 明确要求创建 Git 提交 | `507-closeout` |
| 中断、换会话、压缩上下文或转交 | `507-handover` |

## 纪律

- 一 skill 一动作；相邻能力靠明确产物和路由接力。
- 每个 skill 可独立触发，不强制走完整链路。
- shared skill（共享技能）只定义结果、边界、产物和验收，不写宿主专属调度。
- 正常功能实现不强行包装成专门 skill；按需求、项目规范和宿主能力持续做到验证完成。
- 施工后按需校准实况地图，再独立审查；只有用户明确要求时才创建提交。
