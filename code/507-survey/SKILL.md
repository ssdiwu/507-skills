# 项目勘察（survey）

在项目内先建立共同上下文：定位问题属于哪里，解释当前机制、边界与不确定性，再判断应继续讨论还是转给专门工作流。

## 覆盖什么

- 理解代码、配置、文档权威或运行机制；
- 定位现象、设计想法或概念在项目里的入口与上下游；
- 为讨论建立项目地图和证据范围；
- 根据已确认任务推荐后续 skill（技能）。

不直接修改代码、配置或项目文档；用户明确要求实施时，退出本 skill 并切换到对应工作流。

## 先读上下文

按需读取：

1. 用户给出的文件、目录、符号和错误信息；
2. 项目根 `AGENTS.md`、`README.md` 与 `doc/README.md`；
3. 相关术语表、决策档案和目标目录上下级 `README.md`；
4. 相关源代码、测试与配置。

没有明确路径时先做窄范围搜索，不直接漫扫整个仓库。只能从局部判断时标注“基于当前已读范围”，不把未读实现说成事实。

## 工作流程

1. **判断问题类型**：理解代码、排查现象、设计讨论、实现前对齐，还是项目地图问题。
2. **建立定位**：说明入口、相关模块与文档、上游、下游、关键术语和证据范围。
3. **解释机制**：根据已读证据解释数据流、控制流、状态变化、文档权威、运行方式和边界情况。
4. **提炼焦点**：指出真正需要讨论、核实或决定的事项；能从文件自行回答的不要反问。
5. **推荐路径**：只推荐，不在本 skill 内执行其它动作。

常见路由：

- bug、报错、崩溃或冲突 → `507-repair`；
- 测试本身是任务 → `507-probe`；明确 test-first → `507-formwork`；
- 地图文档缺失或失真 → `507-asbuilt`；完整项目地基巡检 → `507-ground`；
- 行为不变的内部改造 → `507-retrofit`；架构摩擦探索 → `507-inspection`；
- 决策未对齐 → `507-grill`；需求规格 → `507-blueprint`；工单 → `507-workorder`；
- 明确变更范围的交付审查 → `507-audit`；外部参照物调研 → `507-research`。

## 输出结构

按复杂度取舍：

```md
## 一句话定位
<问题在项目中的位置或层次。>

## 已查看的依据
- `<path>` — <相关性>

## 项目地图
- 入口：...
- 相关模块 / 文档：...
- 上游 / 下游：...
- 关键术语：...

## 当前机制
<基于已读证据的解释。>

## 讨论焦点
- ...

## 下一步
- 判断：...
- 推荐：...
- 原因：...
- 先做：...
```

只解释一段代码时，可压缩为：一句话总结 → 调用地图 → 工作原理 → 边界情况，并说明已读依据。

## 红线

- 不直接改文件；
- 不跳过项目文档直接解释源码；
- 不把猜测写成事实或最终方案；
- 不替修复、测试、审查、地图维护或 Git（版本控制）收尾 skill 执行。
