# 画蓝图（blueprint）

把讨论过的上下文沉淀成 **PRD（产品需求文档）**——回答要解决什么问题、给谁、怎么验证。

解决的核心问题是 **misalignment（对齐失败）的后置兜底**——方案问透后，如果不落成一份需求规格，直接进执行，agent 容易按自己的理解跑偏。blueprint 是对齐→执行之间的**可选规格化层**：轻任务跳过它直接执行；重任务或需要留档的需求，先画蓝图再执行。

`507-blueprint` 做的是：**对话/方案 → PRD 需求规格**。

## 覆盖什么

- 从当前对话/方案合成 PRD（不重新访谈，只综合已有信息）
- 把 grill 的对齐结论落成可留档的需求文档
- 为执行提供需求规格基准

## 不覆盖什么

- 开工前把方案问透 → 用 `507-grill`（blueprint 是对齐的下游：先问清要做什么，再画蓝图）
- 把方案拆成可领取的 GitHub issue 工单 → 用 `507-workorder`（issue 是可领取的执行单元，PRD 是需求规格，两件事）
- 写正式汇报型方案 → 不在本 skill 范围
- 架构体检找重构机会 → 用 `507-inspection`
- 执行编排 → 不在本 skill 范围（blueprint 产需求规格，不编排执行）

## 核心纪律（不可违反）

1. **不空访谈**：综合已经说过的内容，不重新采访 507。缺关键事实时，一次只问一个问题，并附推荐答案。
2. **用项目语言**：先读 `doc/术语表.md` 和相关 `doc/决策档案/`；PRD 标题和正文使用项目术语。
3. **测试接缝先想清楚**：写 PRD 时先想"这个功能会在哪个公开接缝验证"；优先已有接缝，必要时才提新接缝。
4. **只写稳定信息**：PRD 默认不塞具体文件路径和代码片段，因为很快过期。例外：原型产出的状态机、schema、reducer、类型形状等能比文字更精确地表达决策时，可截取关键片段并注明来源。
5. **落点是"要做什么/为什么"，不是"怎么改"**：PRD 描述需求和行为，不规定实现步骤。

## PRD 模板

适用：507 说"写 PRD / 把刚才整理成需求 / 形成需求文档"。

流程：

1. 读已有上下文和项目文档，理解当前代码状态。
2. 草拟测试接缝：这个需求完成后，应该通过哪个公开接口或用户路径验证？和 507 快速确认。
3. 写 PRD。

```md
## Problem Statement

从用户视角描述问题。

## Solution

从用户视角描述解决方案。

## User Stories

1. As a <actor>, I want <feature>, so that <benefit>.
2. ...

## Implementation Decisions

- 需要构建/修改哪些 module（模块）或 interface（接口）
- 技术澄清、架构选择、schema/API/交互约定
- 不写易过期的具体文件路径，除非是决策级原型片段

## Testing Decisions

- 好测试验证外部行为，不测实现细节
- 计划在哪些公开接缝测试
- 代码库里已有的相似测试先例

## Out of Scope

明确不做什么。

## Further Notes

补充说明、风险或开放问题。
```

## 和其他 skill 的边界

| skill | 分工 |
|---|---|
| **blueprint** | 对话/方案 → PRD 需求规格（本 skill，想清楚要做什么） |
| **workorder** | 任务 → GitHub issue 工单（blueprint 的可选下游；PRD 想清楚后，拆成可领取工单） |
| **grill** | 把方案/改动问透（blueprint 的上游） |
| **执行入口** | 编排执行（blueprint 产需求规格，不编排执行） |

时序：对齐 → blueprint 画 PRD（可选）→ workorder 开工单（可选）→ 执行。

blueprint 和 workorder 的区别：**PRD 是"想清楚要做什么"（需求规格），issue 是"把要做的事写成可领取的工单"（执行单元）**。它们是两件事——PRD 给项目自己看，issue 给 GitHub 上的 AI/协作者领取。重任务两者都走（先 PRD 再拆工单），轻任务都可以跳过直接执行。

## 启动姿势

当 507 说"写 PRD""把刚才整理成需求""形成需求文档"时：

1. 读已有上下文和 `doc/术语表.md`，用项目语言。
2. 草拟测试接缝并和 507 确认。
3. 按 PRD 模板写需求规格。
4. 提示下一步：需要拆成可领取工单 → `507-workorder`；直接执行 → 交执行入口。

## 红线

- 不拆 issue 工单（归 `507-workorder`）。
- 不写实现步骤（写需求和行为，实现走执行入口）。
- 不空访谈（综合已有上下文）。
- 不塞易过期的文件路径/行号（耐久优先）。
- 不在和 workorder 的边界上抢"开工单"的活。
