# 架构体检（inspection）

定期给代码库做架构体检，找出**加深机会（deepening opportunities）**——把"接口和实现一样复杂"的浅模块，改成"小接口藏大量行为"的深模块。目标是**可测性和 AI 可导航性**，对抗 agent 加速带来的软件熵增。

建议每隔几天跑一次。

## 覆盖什么

- 已有代码的架构体检（找浅模块、找该合并的碎模块、找难测的部分）
- 把摩擦点排成可执行的候选列表
- 选定一个候选后，拷问设计 + 对比接口方案

## 不覆盖什么

- 开工前的轻量对齐 → 不在本 skill 范围
- 落成方案文档 → 不在本 skill 范围
- 实际改代码 → 拷问完交正常开发流程（如需后台多角色实施可走 dteam）

## 核心词汇（本 skill 全程复用，别漂移成 component/service/API/boundary）

- **Module（模块）**：有接口+实现的东西，尺度无关（函数/类/包/切片都算）。
- **Interface（接口）**：调用者要知道的**一切**——类型 + 不变量 + 错误模式 + 顺序 + 配置。不只是类型签名。
- **Depth（深度）**：单位接口能调动的行为量。小接口藏大行为 = **深**；接口≈实现复杂度 = **浅**。
- **Seam（接缝）**：不就地改就能改变行为的位置（出自 Michael Feathers）。说接缝，别说 boundary。
- **Adapter（适配器）**：在接缝处满足接口的具体物。
- **Leverage（杠杆）**：调用者从深度获得——一次实现，N 处复用。
- **Locality（局部性）**：维护者从深度获得——改一处，bug 集中。

## 三个判断工具（贯穿全程）

1. **删除测试**：想象删掉这个模块。复杂度消失 = 它只是透传（该删）；复杂度在 N 个调用处重新出现 = 它在真正减负（留）。→ 判断模块值不值得存在。
2. **接口即测试面**：调用者和测试跨同一个接缝。想测接口内部 = 模块形状错了。
3. **一个 adapter = 假设接缝，两个 = 真接缝**：没真正变化就别引入接缝。（这条已进 AGENTS.md，这里复用。）

## 流程三步

### ① 探索：找摩擦点（主 LLM 自己做，不用子进程）

先按 AGENTS.md 文档优先级读 `doc/术语表.md` 和 `doc/决策档案/`，掌握现有术语和已记录决策——**别重复审议已有 ADR**。再探索代码，找以下信号：

- 懂一个概念要在很多小模块间反复跳。
- 模块**浅**——接口和实现一样复杂。
- 为了测试硬抽了纯函数，但真 bug 藏在"它怎么被调用"里（没有 locality）。
- 紧耦合模块跨接缝互相泄漏。
- 哪些部分没测，或按当前接口很难测。

对可疑的浅模块做**删除测试**。别机械套规则，跟着真实摩擦走。

### ② 出报告：候选列表（默认 markdown，可选 HTML）

把摩擦点排成候选报告。**默认 markdown**（终端友好、可贴文档、零依赖）；用户说"出 HTML 报告"时才生成 HTML。

每个候选一张卡，包含：

- **文件**：涉及哪些文件/模块
- **问题**：当前架构为什么造成摩擦
- **方案**：白话描述改成什么样
- **收益**：用 locality 和 leverage 解释，测试会怎么变好
- **Before / After 对比**：markdown 用表格或文字描述；HTML 版才画 Mermaid/SVG 可视化
- **推荐强度**：`Strong` / `Worth exploring` / `Speculative`

报告结尾给一个 **Top 推荐**：先做哪个、为什么。

**关键纪律**：这一步**只提摩擦点，不提具体接口方案**。报告写完问用户："想深入哪个？"

**术语要求**：领域用 `doc/术语表.md` 的词，架构用本 skill 词汇。术语表 定义了"Order"，就说"Order 接单模块"，别说"FooBarHandler"或"Order service"。

**ADR 冲突**：候选若和现有 ADR 矛盾，只在摩擦真实到值得重开 ADR 时才提，并在卡片标明（如警告："与 ADR-0007 冲突——但值得重开，因为…"）。

**HTML 报告可选细节**：写到系统 temp 目录，不污染 repo——解析 `$TMPDIR`（回退 `/tmp`），文件名 `architecture-review-<时间戳>.html`，生成后用 `open <绝对路径>` 打开（macOS）。Tailwind + Mermaid 走 CDN。每个候选配 before/after 可视化。

### ③ 拷问选定候选（调用 507-grill）

用户选定一个候选后，进入拷问设计树——约束、依赖、加深后模块的形状、接缝背后藏什么、哪些测试活下来。**这一步直接调用 `507-grill`**：一次一个问题、附推荐答案、能查代码就别问、走遍决策树。

边拷问边沉淀（同 grill）：
- 命名加深模块时，若用到 `doc/术语表.md` 没有的概念 → 当场加进术语表（惰性建文件）。
- 锐化模糊术语 → 当场更新 `doc/术语表.md`。
-用户用承重理由否决候选 → 按三门槛判断要不要记 ADR（参考 AGENTS.md 决策记录准则）。

需要探索接口的多个替代方案时，走下一步。

### ④ 接口并行脑暴（可选，用 pi 自递归）

选定加深候选后，若需要对加深后的模块做"Design It Twice"（出自 Ousterhout——第一个想法往往不是最优），按 `references/brainstorm-recipe.md` 的配方并行 spawn 3 个 pi 子进程，各自设计截然不同的接口再对比。配方、对比维度、参数说明都在该文件，本文件不重抄。
