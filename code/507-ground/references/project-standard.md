# 项目标准规范

这份规范适用于 `/Users/diwu/Documents/codes/` 下的主动维护型代码项目。

目标是让任何 agent 进入一个项目时，先能找到：入口、文档地图、项目语言、关键决策，以及各代码目录的职责说明。

## 必备项

### 1. `AGENTS.md`

项目级 agent 规范文件，必须存在。

应包含：

- 项目一句话定位
- 先读文档顺序
- 代码边界与安全边界
- 验证方式
- 项目特有禁忌或不做事项
- **代码工程纪律**（适用于含代码的项目，由 `507-ground` 写入）：
  - **删除测试判断模块价值**：判断一个模块/抽象是否值得存在，想象删掉它——复杂度消失说明它只是透传（删）；复杂度在多个调用处重新出现，说明它在真正减负（留）。
  - **接缝纪律**：只在真有变化的地方引入接口/抽象层。只有一个实现（adapter）的是“假设接缝”，两个以上不同实现才是真接缝；别为单一用法提前抽接口。
  - **函数粒度**：函数控制在 100 行以内；超出则考虑拆分。
  - **测试看行为**：测试优先通过公共接口验证行为，不测内部实现；mock 只放在系统边界。
  - **先建反馈环再调 bug**：调 bug 先造一个快速、确定性、agent 能跑的 pass/fail（成败）信号（失败测试/curl/CLI 重放/headless 等）；没有反馈环就别盯着代码空猜，列已试方法后求助用户。信号是 90% 的调试，其余是机械操作。
  - **插桩打 tag**：所有临时 debug 日志打唯一前缀 tag（如 `[DEBUG-a4f2]`），清理时一个 grep 全删；未打 tag 的临时日志会残留。

不应包含：

- 冗长的背景故事
- 大段重复文档地图
- 术语表正文
- 架构决策全文

### 2. `README.md`

项目入口说明，必须存在。

应回答：

- 这是什么项目
- 给谁用
- 怎么运行
- 目录怎么分
- 主要入口在哪

### 3. `doc/`

项目知识目录，必须存在。

最少应有 `doc/README.md`。

`doc/README.md` 应该说明：

- 这套文档有哪些
- 先读什么、后读什么
- 哪些是当前实现权威
- 哪些只是历史归档

主动维护型项目默认采用编号目录组织 `doc/`，除非项目很小（少于 5 个文档）或已有清晰历史组织方式：

| 段 | 职责 | 权威性 |
|---|---|---|
| `00-产品与原则/` | 产品定位、目标用户、体验原则、UI 气质 | 长期有效权威 |
| `10-架构与运行/` | 当前架构、代码现状、运行、测试、开发规范 | 当前实现权威 |
| `20-能力参考/` | 外部能力事实库、调研、实测数据、license、能力边界 | 事实参考，不决定排期 |
| `30-路线图/` | 版本维度编排、后续计划 | 路线图唯一权威 |
| `40-版本实施方案/` | 版本级落地方案，必须含 checklist | 受路线图约束 |
| `90-归档/` | 已完成、过时或仅用于追溯历史背景的方案 / 日志 | 否，仅查历史事实 |

不是每个项目都必须创建所有编号目录；只创建有实际内容的段。若已有路线图或版本实施方案，就不应继续平铺在 `doc/` 根目录，应放入对应编号段。

## 条件必备项

### `CHANGELOG.md`

当项目有版本发布、对外用户或多开发者协作时，应建立 `CHANGELOG.md`，记录每个版本的可见变更（功能、修复、破坏性变更、迁移说明）。

适用：对外发布的库、CLI、SDK、应用；有版本号标签的项目；多人协作的项目。
不强制：纯个人工具/实验仓库、内部脚本、一次性原型。

写法要求：

- 按版本倒序，最近版本在最上。
- 每条变更用**用户/使用者能感知**的语言，不写"重构了内部模块 X"。
- 区分类型（Added / Changed / Fixed / Removed / Breaking），保持格式一致。
- 破坏性变更必须显式标注，并给出迁移说明。
- 不写未发布的内部计划；路线图归 `doc/30-路线图/`。

不写：

- 提交粒度的完整 git log
- 实现细节
- 临时方案

### `doc/术语表.md`

当项目存在稳定的领域词、系统词或经常混用的概念时，应建立 `doc/术语表.md`。

它只做一件事：定义“这个项目里这个词精确指什么”。

写法要求：

- 定义写“它是什么”，不是“它做什么”。
- 同一概念只保留一个规范叫法。
- 其它常见但不推荐的叫法，放进 `_Avoid_`（禁用同义词 / 别名）。
- 只收项目语言，不收通用编程概念、模块名或类名，除非它们本身就是领域词。

不写：

- 实现细节
- 临时方案
- 决策讨论过程
- 草稿内容

### `doc/决策档案/`

当出现同时满足以下三条的决定时，应建立 `doc/决策档案/` 并记录：

1. 难逆转
2. 没有上下文会让人困惑
3. 存在真实权衡

如果只是小约定、临时实现或普通笔记，不进 ADR。

文件与索引规则：

- **文件命名**：`0001-中文标题.md`。编号前缀保顺序，标题用中文描述决策主旨；项目专有术语沿用 `doc/术语表.md` 的规范叫法（不强行中英互译），如 `0006-建检循环与三层结构.md`。
- **索引**：维护 `doc/决策档案/README.md`，每条一行：编号 + 标题 + 一句话主旨（写「定了什么」不写「为什么」，不超过一行）。新增或更新 ADR 时同步索引；索引失真即视为文档债。
- **何时建索引**：第一条 ADR 落地时就建 `README.md`（1 条也建）。`doc/README.md` 的决策档案那行只指向 `doc/决策档案/README.md`，不再逐条罗列，避免两处重复维护。

### `doc/经验笔记.md`

收"可改的做法与避坑经验"，回答"这事儿怎么做"。与术语表、决策档案按边界分工：术语表收词，决策档案收不可逆决策，经验笔记收可改做法。

**准入门槛**：解决一个坑时，如果换一个无上下文的 agent 来会重走一遍，就值得记。不满足 ADR 三条、也不是词的定义，但下次还会用到的经验走这里。

**条目格式**：现象 + 做法 + 证据（日志路径 / commit / ADR 指向）。

**维护**：重复发生时在原条目追加证据；做法被推翻就改这条。惰性创建，没值得记的经验就不建空文件。

### 含代码目录的 `README.md`

任何包含代码的目录都应有自己的 `README.md`，尤其是：

- `src/`
- `tests/`
- `scripts/`
- `bin/`
- `packages/`
- `apps/`
- 其它项目特有代码目录

## 命名约定

- 项目术语表统一使用 `doc/术语表.md`
- 不再默认使用 `glossary.md`
- 如果项目历史上已有 `docs/`，优先迁移到 `doc/`，或明确说明为何保留

## 项目初始化顺序

推荐按这个顺序落地：

1. `AGENTS.md`
2. `README.md`
3. `doc/README.md`
4. `doc/` 编号目录地图（按需创建 `00 / 10 / 20 / 30 / 40 / 90`）
5. `doc/术语表.md`
6. `doc/决策档案/`
7. `doc/经验笔记.md`（惰性，有经验才建）
8. 各代码目录 `README.md`

## 通过标准

一个项目只有在下面这些点上同时成立，才算符合标准：

- agent 进来能知道怎么读
- 人进来能知道这是什么、怎么跑
- `doc/` 能区分当前实现、路线图、版本实施方案和历史归档
- 项目语言有统一叫法
- 关键决策能找到出处
- 代码目录不会失去职责说明

## 例外

只有用户明确说明时，才允许偏离这份标准。
