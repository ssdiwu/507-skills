# Agent 行为规范

## 1. 身份与沟通

- 默认使用中文沟通；称呼遵循用户偏好或项目约定，不预设个人昵称。
- English（英文）技术术语、命令、路径、文件名、代码标识符、工具名出现时，使用 `English（中文解释）` 格式。
- 示例：`README.md`（说明文档）、`Git commit`（Git 提交）、`API`（接口）、`path`（路径）。
- 先给结论，再给必要细节；避免空泛复述。
- 阻塞时直接说明 blocker（阻塞点）、影响范围和下一步建议。
- 能自己查代码、文档或现有上下文回答的问题，不要反问用户。
- 必须向用户澄清时，当前已知、互不依赖且有合适收集工具的决策可批量询问；其余一次只问一个问题，并附上推荐答案。
- 表达默认简洁，去掉废话和重复；但技术事实、边界和安全提示不能省。

## 2. 工作方式

- **先读后写**：先看文件、搜索现状、确认上下文，再做判断或改动。
- **新任务最小必读顺序**：处理一个项目的新任务前，至少先读项目根 `README.md`（说明文档），再读 `doc/README.md`（文档总览）。这是默认最小上下文；
- **文档优先级**：查找信息时按以下顺序搜索，找到即停，不要跳级：
  1. **`doc/` 目录** — 架构文档、设计决策、规范定义（最高优先，先查这里）
  2. **`src/` 各层 `README.md`** — 代码层说明
  3. **源代码本身** — 具体实现细节
  - **禁止一上来就搜 archive 或 grep 源码，必须先过 doc/**
- **决策记录准则**：设计决策只有同时满足“难逆转、无上下文会让人困惑、有真实权衡”三条，才记入 `doc/决策档案/`（架构决策记录）；其余不记，避免决策文档膨胀。对齐或讨论中确定的术语当场写入 `doc/术语表.md`，不批量补。
- **三次原则**：如果一件事还没有重复第三次，不要过早抽象；三行重复优于一次抽象；没有实际内容不建目录、文件或抽象层，惰性创建。
- **垂直切片**：拆计划、issue（问题单）或实现步骤时，优先按可独立验证的端到端路径切；避免按 schema/API/UI 等水平层拆。
- **未验证不算完成**：改完必须运行测试或检查输出；无法验证时说明原因和后续验证方式。
- **测试分层主动跑**：完成任何功能改动后，按项目现有测试分层主动跑测试并报告结果（单元 → 集成 → 端到端）；项目缺某层测试时说明原因，不跳过不报；测试失败先按 `507-fix` 建反馈环，不直接改断言。
- **遵循现状**：代码风格、命名、注释、目录结构优先跟随项目已有约定。
- **最小实现梯子**：充分理解任务与受影响链路后，依次检查“是否真需要 → 项目已有实现 → 标准库 → 平台原生能力 → 已安装依赖 → 最小新代码”，在第一个能完整满足需求的层级停止，不为未来可能性继续扩写。不得借最小化省略明确需求、输入校验、防止数据丢失的错误处理、安全、无障碍或项目要求的测试与验证。
- **价值优先**：不为“更完整 / 更优雅 / 更好看”而优化；只有当改动能明确提升成功率、稳定性、理解准确度、可验证性或显著降低用户成本时，才做优化。
- **先证据后优化**：做优化前，先说明它对应的真实问题、失败样例或验证假设；没有证据时，优先做最小验证，而不是继续扩写 prompt、schema 或抽象层。
- **安全优先**：绝不硬编码密钥或 `API Key`（接口密钥）；校验用户输入；显式处理错误。

> 以下代码工程纪律适用于代码项目，由 `507-setup` 写入各代码项目自己的 `AGENTS.md`，不在此全局文件重复：删除测试判断模块价值、接缝纪律（一个 adapter 是假设接缝）、函数粒度 100 行、测试看行为不看实现、mock 只放系统边界、先建反馈环再调 bug、插桩打唯一 tag。

## 3. Skill 使用约定

- 通用 skill 放在 `~/.agents/skills/`（共享技能目录），由兼容 Agent Skills 标准的宿主按 `description` 自动发现；需要强制时使用宿主原生显式调用语法。
- 不确定从哪开始时，默认用 `507-explore` 只读梳理项目语境、已知、未知与当前 `frontier（可推进边界）`。实施完成后用 `507-review` 做独立审查；只有用户明确要求创建 Git 提交时才用 `507-commit`。中断、换会话、压缩上下文或转交时用 `507-handoff`。
- **普通新任务 / 需要持续做到完成**：使用当前宿主可用的执行与验证能力持续推进，不停在分析、TODO 或半截进度；宿主专属调度规则写在该宿主自己的 `AGENTS.md`，不写入共享 skill。
- **`507-explore`**：面对尚未看清的项目问题，读取既有共识与现实证据，在会话中累积探索快照并路由；只读，不创建持久探索地图。
- **`507-explain`**：把用户不理解的单个概念讲懂；关系复杂时选用表格、`Mermaid（图表语法）`或 `ASCII（字符图）`示意，默认不落盘。
- **`507-grill`**：实施前对齐真实用户决策；一次只推进一个依赖层，独立问题可批量收集，并同步沉淀够格的术语与决策。
- **`507-fix`**：已明确是 bug、报错、崩溃、回归或合并冲突时，先建反馈环，再做最小根因修复。
- **`507-test`**：测试本身是主任务时，补行为测试、运行测试、修正过时测试或缩小失败范围。
- **`507-tdd`**：只有用户明确要求 TDD（测试驱动开发）、test-first（测试先行）或红绿重构时使用。
- **`507-simplify`**：保持公开契约和可观察行为不变，直接处理指定范围，或逐项处理 `507-inspect` 的完整报告。
- **`507-map`**：以当前代码为证据维护根/分层 README、doc 组织、目录职责和项目地图；只改地图文档。
- **`507-review`**：审查任意明确变更范围的项目规范、需求符合度和代码质量；只报告与路由。
- **`507-commit`**：用户明确要求 commit（提交）时，轻量复核、同步必要产物、精确暂存、验证并创建真实本地提交；默认不 push。
- **`507-handoff`**：中断、换会话、转交或压缩上下文时，只在对话中输出脱敏交接摘要。
- **`507-setup`**：项目初始化或全量规范巡检；核对文档与代码现实，先报告再路由，不修改业务代码。
- **`507-prd`**：把当前对话或方案沉淀成 PRD（产品需求文档）。
- **`507-issue`**：把任务结构化成可上传 GitHub、可被 agent 领取执行的 issue（问题单）。
- **`507-prototype`**：正式实现前用可丢弃原型验证一个具体假设。
- **`507-inspect`**：只读架构审查，产出带代码证据和置信等级的完整报告，再交给 `507-simplify`。
- **`507-research`**：研究外部项目、文章、竞品或参考实现的真实机制与借鉴意义。

## 4. 项目标准与 README 同步

- 主动维护型代码项目必须从开局具备项目级 `AGENTS.md`、根 `README.md`、`doc/README.md` 与 `CHANGELOG.md`（变更日志）；后者采用 [Keep a Changelog](https://keepachangelog.com/zh-CN/1.1.0/) 基本格式，以 `## [Unreleased]`（未发布）起步，不为初始化倒填历史提交。
- 项目术语统一沉淀到 `doc/术语表.md`。
- 只有符合“难逆转、无上下文会让人困惑、有真实权衡”的决策才写入 `doc/决策档案/`，不为空规范创建空目录。
- **三个沉淀出口**（术语表 / 决策档案 / 经验笔记，按边界分工，不混用）：
  - **`doc/术语表.md`** — 回答“这个词指什么”，收项目特有概念；定义“是什么”，不沾实现细节。惰性创建，术语敲定时当场写。
  - **`doc/决策档案/`** — 回答“为什么这么定”，只收“难逆转 + 无上下文会困惑 + 有真实权衡”的决策（刻碑，记了就不删）。一条一文件，顺序编号 `0001-中文标题.md`（项目术语沿用术语表规范叫法）；维护 `README.md` 索引（编号 + 标题 + 一句话主旨），新增 / 更新 ADR 时同步。
  - **`doc/经验笔记.md`** — 回答“这事儿怎么做”，收可改的做法与避坑经验（活页）。门槛：解决一个坑时，如果换一个无上下文的 agent 来会重走一遍，就值得记。格式：现象 + 做法 + 证据。重复发生时在原条目追加证据，不新建条目。
- 凡包含代码文件的目录，必须有 `README.md`（说明文档）。
- 若代码按子目录嵌套组织，每一层包含代码的目录都应有自己的 `README.md`。
- 改代码前，必须阅读目标文件所在目录及相关上下级目录的 `README.md`。
- 改代码后，如果职责、结构、入口、运行方式或约定变化，必须同步更新对应 `README.md`。
- 发现 `README.md` 缺失时，先补齐最小必要说明；无法补齐时说明原因和影响范围。

## 5. Git 规范

- 每次 `Git commit`（Git 提交）只做一件事，精准 `git add`（添加到暂存区）相关文件。
- 遵循 `Conventional Commits`（约定式提交）：英文类型前缀 + 中文描述，格式 `type: 中文描述`。除非仓库已有强制规范，否则默认用 `feat` / `fix` / `refactor` / `docs` / `test` / `chore` 等英文前缀。
- 示例：`fix: 修复术语表文件名同步`、`refactor: 收敛 Git 提交规则`、`docs: 补决策档案目录职责`。
- 提交前检查变更范围，避免混入无关改动。
- `git commit`（Git 提交）前确认 `CHANGELOG.md`（变更日志）的 `Unreleased`（未发布）段是否反映本次用户/使用者可感知变更；用 `Added` / `Changed` / `Fixed` / `Removed` / `Breaking` 分类，不写实现细节、内部计划或逐提交日志。纯内部重构、测试或文档地图调整不强塞条目。
- 对外发布时，版本号、`CHANGELOG.md` 与 `Git tag`（Git 标签）一致；发布提交后按项目惯例打 tag（默认 `v<x.y.z>`），推送时提交与 tag 一并推送。
- 禁止 `git push --force`（强制推送）到 `main`（主分支）。

## 6. 循环防护

- 对同一文件第 3 次搜索、连续 3 次重复同类命令、连续 3 次说同样的话时，立即停止并换方法。
- 禁止渐进式扩大参数，例如 `grep -A 60 → 70 → 80`。
- 小文件直接全文读取；大文件先看结构，再精准搜索。
- 连续 5 个工具调用没有新信息，或连续 3 次卡在同一问题上，立即说明现状并询问下一步。
