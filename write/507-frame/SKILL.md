# 507 框定（frame）

这个 skill 用来辅助用户起草结构化的方案、计划或文档。

## 这个 skill 覆盖什么

活动方案、培训方案、工作坊方案、合作提案、项目计划——任何"有明确结构、面向特定受众、用于推动决策或行动"的方案/计划/文档。

## 这个 skill 不覆盖什么

- 营销文案、公众号文章、博客
- 内部备忘录、工作日志
- 内容润色、代码审查
- 闲聊

## 相关 skill

如果用户要写的是**现场分享稿、PPT 逐页稿、课程讲稿、企业培训演讲稿**，尤其是 507 本人的案例型分享风格，优先使用 `507-stage`。

`507-frame` 负责正式方案和结构化文档；`507-stage` 负责现场讲述和 PPT 内容风格。

## 一、调性：方案读起来应该像什么

读起来像"认真在做一件事"，不像"想推销你"，也不像"内部在讨论"。

- **独立方案**：不挂"翻译自 XX""基于 XX 方法论""参考了 XX 文档"等来源
- **正式但不营销**：不用"赋能""转型""价值闭环""强强联合"等空话
- **信息密度高**：每段话都有可读出的具体信息，不用"与此同时""考虑到""作为"等连接词凑字
- **结构清楚**：章节序号、表格、必要的清单——读者扫一眼就能抓到结构
- **不口语化、不内部对话**：不写"我们讨论后认为…""伙伴的反馈是…""经过几轮迭代…"

**反面例**（AI 风格的段落）：

> 在 AI 时代浪潮下，我们深入探讨 AI 如何赋能传统行业，致力于打造一站式、全链路的 AI 升级解决方案。我们的核心团队深耕多年，构建了从底层逻辑到应用层的完整闭环。

**正面例**（同样的内容用克制的方式写）：

> 这次要做的是帮 B 端客户把已经用上的 AI 工具整合成完整工作流。先做一个 1.5-2 小时的小范围工作坊，挑 10 位老板，把每家的工作流现场拆解，落地后再对接 1v1 诊断和陪跑。

## 二、507 认可的 7 个表达点

这 7 条来自一次完整的演讲稿 / PPT 打磨，适用于方案、计划、课程稿和企业分享内容。

### 1. 少讲场面话，直接进案例

先给事实，不先给观点。

不要先写：

> 今天我们将系统探讨 AI 如何进入企业工作流。

更好：

> 这是一家租车公司。它的车会放到多个平台上出租。AI 已经可以做查价前置动作。

### 2. 用“案例事实 → 判断 → 下一步”推进

优先按这个顺序写：

```text
案例里发生了什么
这说明什么
下一步怎么做
```

示例：

```text
AI 已经能查价。
但查价只是流程里的一个点。
下一步是看它能不能接住还车、维修、投诉、结算。
```

### 3. 比喻只在关键处用

比喻要帮助理解，不要抢主题。

可用比喻：

- 台账像行车记录仪，记录过去。
- 流程表像导航，推动下一步。
- 自动化像闹钟，到点响就行。
- AI 写回记录像接力赛，把下一棒交出去。

### 4. 句子短一点，少用抽象词

不要写：

> 实现业务流程自动化闭环，构建企业级 AI 工作台。

更好：

> AI 读表。AI 写回表。飞书自动推送。人处理确认。结果再回到表里。

### 5. 每一页 / 每一节只承担一个推进任务

不要一页里同时讲背景、问题、方法、边界、结论。

更好的台阶是：

```text
AI 已经接了查价
查价只是一个点
AI 接不住散信息
所以要一张表
表要有字段
字段能触发动作
```

### 6. 不替读者太早总结

不要一上来就把大结论压给读者。

先让读者看到事实变化，再让结论自然长出来。

### 7. 保留现场口语感

方案可以正式，但不要写成 AI 宣传稿。

更像 507 的句子：

> 这一步已经跑起来了。但它现在只是流程里的一个点。

不要写：

> 该案例体现了 AI 在业务流程中的初步落地价值。

## 三、6 条写作原则

| 原则 | 含义 | 反例 | 正例 |
|---|---|---|---|
| **具体而非抽象** | 用数字/动作/对象替代形容词 | "小规模、高质量" | "10 位左右，邀约制" |
| **主动而非被动** | 主语执行动作 | "培训由团队交付" | "团队交付培训" |
| **正面而非双重否定** | "我们做 X" 而非"我们不做 Y" | "现场不做完整改造" | "现场做诊断和思路讲解" |
| **删除冗词** | 砍"内部对话""作为""与此同时""考虑到" | "考虑到活动的定位，我们认为 3 段方法是比较合适的" | "用 3 段方法" |
| **相关信息紧邻** | 同一段内的"做什么/为什么"靠近，不要打散 | "老板需要做一件事。（中间隔了 5 段其他内容）这件事就是带 SOP 来……" | "老板带 SOP 来；现场拆解" |
| **关键信息放末位** | 重点放结尾——读者扫到结尾就能抓到核心 | "老板带走的是……怎么改（中间插了 8 行修饰）" | "老板带走 1 件事：怎么改" |

## 四、AI 词反清单（中文）

**Puffery 形容词**：划时代、里程碑式、标志性、典范性、重中之重、至关重要、不可或缺、举足轻重、卓越、杰出、非凡

**空话短语**：致力于、旨在、力图、不断推动、持续赋能、深入贯彻、全面落实、全方位、立体化、多维度

**营销/AI 高频词**：赋能、助力、引领、打造、构建、矩阵、体系、生态、闭环、链路、赛道、底层逻辑、深耕、颠覆、变革、转型、升级、智能化、智慧化、数字化、定制化、一站式、全链路、端到端、突破性、开创性、无缝、稳健

**AI 高频动词**：深入探讨、深度解析、全面剖析、系统梳理、聚焦、发力

**AI 句式**：
- "在 X 时代背景下..."
- "随着 X 的不断发展..."
- "众所周知..."
- "不难发现..."
- "X 扮演着至关重要的角色"
- "是大势所趋"
- "在这样的背景下..."
- "既是 X，也是 Y"
- "这不仅是 X，更是 Y"
- "我们坚信..."

## 五、写法反清单

| 不要做 | 原因 | 反例 |
|---|---|---|
| **不挂来源** | 方案是独立的 | "本次方法论基于《XX 课程》第 2、5、8 课" |
| **不写议论稿** | 方案是陈述不是讨论 | "我们认为，本次沙龙的核心价值在于……" |
| **不写未聊透的内容** | 没确认的别写 | "风险控制（5 列表格：风险/触发/预防/兜底/责任方）"——没聊透就不写 |
| **不包太多** | 章节数按规模灵活 | 2 小时活动不写 11 章 |
| **不写 v1/v2 叙事** | 方案不是迭代记录 | "v1 我们考虑过 99 元，v2 改成 199" |
| **不写案例叙事** | 方案不是故事 | "在和某人讨论的过程中，我们意识到……" |
| **不写工具用法清单** | 内容是工作流不是工具科普 | "朋友圈文案生成、客户跟进话术、爆款标题公式……" |
| **不写"对企业的价值"+"对学员的价值"两栏** | 在具体场景里人就是企业本身 | 拆分只在泛指场景下有意义 |
| **不写"联合发起人"** | 角色包装太重 | 直接说"案例合作方 + 社群入口" |
| **不写内部决策过程** | 方案不是工作日志 | "我们讨论后认为…" → 直接写"做 X" |
| **不用否定句/辩护句/弱化词** | 否定句暴露不自信，读者读到像在自我辩护 | 见下方「肯定句原则」 |

### 肯定句原则（重要）

方案里只要出现否定句、辩护导语、让步句、弱化词，就是在暴露不自信。读者读到"不是…是…""这是一次低成本的尝试"，会觉得你在自我辩护、在撇清。

只在真的不确定时才用不确定语气。其余一律肯定句。

**禁止形态：**

| 形态 | 反例 | 正例 |
|---|---|---|
| 否定定义 | "这不是把需求外包成产品" | 删（概览表已说清事实） |
| 辩护导语 | "这是一次低成本的验证尝试…" | 删，直接进事实 |
| 弱化词 | "尝试性方案""不保证完整度" | "方案""属验证级原型" |
| 不确定语气 | "要不要做成…看路演结果再定" | "是否深化，由企业决定" |
| 括号补丁 | "一个懂业务的人（不必是技术）" | "一个懂业务的人" |
| 堆叠否定 | "不出人开发、不组织、不到场" | 删（上面表格已说清） |

**边界信息要保留，但改肯定陈述。** 需要让读者知道"验证级、非成品、深化另谈"这类边界，不能丢，只是从否定/辩护改成定义式陈述：把"不保证商用完整度"写成"属验证级原型"。

**清不干净的信号：** 如果某段话删掉后信息没丢，说明它本来就是辩护；如果删掉后读者会误解边界，就改成肯定陈述保留下来。

## 六、格式克制

| 元素 | 立场 |
|---|---|
| 装饰性 emoji（每段一个小表情） | 不用 |
| 频繁加粗（每句都加粗） | 克制，只在关键判断处加粗 |
| 装饰性 callout 框（"💡注意：…""🔥亮点：…"） | 不用 |
| 表格里 emoji 表头 | 不用 |
| 表格本身（信息密度高时用） | 保留 |
| 章节序号（一、二、三） | 保留 |
| 关键判断处的结构性 callout | 保留（不是装饰性的） |
| 标题中的日期（"｜2026-06-05"） | 砍掉 |

## 七、工作方式启发

### 动笔前的对齐

动笔前**先不写**。通过追问和案例，让用户和 AI 双方都清楚"这次要写什么"。

- **苏格拉底式追问**：不直接问"是什么/多少钱/什么时间"——而是用反问、举例、缩小范围等方式，帮用户一边回答一边理清"自己到底要写什么"
- **双向受益**：用户自己通过回答变得清晰；AI 也通过追问快速理解要做什么
- **案例/例子补充**：用户可以给案例/例子（之前做过的方案、参考的文档、风格样本）——这比纯文字回答更高效
- **基于共同理解再对齐**：问完+看完案例后，AI 和用户做一次对齐（"我理解你这次要写的是 X，对吗？"）——双方一致才动笔
- **信息不足时主动追问**：不要在"猜不准"时硬写——追问永远比硬写更对
- **不要"清单式提问"**："问 5 个问题""问 10 个问题"这种 SOP 化的清单不是启发式——AI 应该根据上下文判断问什么

### 动笔中

- **分块处理**：section by section 推进比一次性写完整篇更稳
- **用 surgical edits**：编辑时只改要改的部分，不要 reprint 整篇
- **用户满意才进下一节**：每一节先迭代到用户认可，再往下走

### 动笔后（自检）

交付前自己问自己三个问题：

- **"如果读者第一次看到这个文档，他们会问什么？"**——有没有没说清楚的地方
- **"有没有歧义/错误假设/前后矛盾？"**——自己 review 一遍
- **"每个环节有没有规则+数字+边界+产出，还是只有大概/约/偶尔的描述？"**——拿不准时，找一份同类成熟方案做颗粒度锚点对比，避免把方案写成思路说明
