# 融合（fuse）

写作工作流的**中间层**：把碎片融合成候选 idea。

解决的核心问题是 **有了碎片和争点材料，但它们提供的解释可能彼此竞争；要写成一篇，不能先锁定结论再挑证据，而要先比较哪些材料能组成不同且成立的文章路径**。fuse 做的是这个组合判断，但产物是**候选**（经 `507-grill` 组织用户审核），不是定稿。

`507-fuse` 做的是：**碎片 + 争点材料 → 比较可竞争解释 → 产 2～5 个候选 idea 交审**。

## 覆盖什么

- mine 产出了一批碎片（本次新落 + 既有），用户想看"这些能不能成题"
-用户直接提供几个已有碎片，问"这几个组合起来够不够写一篇"
- 定期回看 `01-碎片/`，发现某主题碎片聚拢到够成题，主动提示融合机会

产物是**候选 idea**，落 `02-创意/候选/`，由用户比较并选择哪条论证组合值得进入正式创意。存在真实竞争解释时产出 2～5 个候选；材料只支持一条路径时可以只产 1 个，不为凑数制造伪分歧。

## 核心纪律（不可违反）

1. **检索与材料核对是必做前置**——融合前必须翻看 `01-碎片/`（本次新碎片 + 既有碎片），并读取 mine 交接的核心争点与来源锚点。理由：每个候选 idea 都默认要检查“能否融合既有碎片”和“是否遗漏了竞争解释”——除非确实没有合适的，否则必须融合。
2. **融合是首选，不硬凑是底线**——有合适的既有碎片就融合；确实没有也不硬嫁接（硬嫁接会稀释判断）。这呼应 `[[三次原则]]`、`[[不是为了好而好]]`。
3. **先比较解释，再挑骨架**——以核心争点为单位比较支持证据、反例、替代解释、边界/缺口与时间/制度语境；再按时效性、主体性和张力挑骨架。反例不是默认淘汰材料，也可能让同一材料组合出另一条文章路径。
4. **产 2～5 个候选，不替用户拍板**——存在真实竞争解释时，融合结果产出 2～5 个**候选 idea**进 `02-创意/候选/`，携带候选进入 `507-grill` 对齐“哪条文章路径成立、想写哪条”；材料只支持一条路径时明确说明并只产 1 个。agent 不得为凑数制造伪分歧，也不得把 skill 名称当成用户选择题。
5. **不是每个碎片都要立刻进 fuse**——只有“几个碎片和争点材料聚拢到够格写一篇”时才进 fuse。拆出来但还不够格的碎片，先沉淀在 `01-碎片/`，等下次有新材料再融合。不“拆一个碎片就硬造一个 idea”。
6. **不写正文**——正文归 forge。fuse 只产候选 idea（选题卡形态），不列完整文章大纲、不写开头。
7. **“落卡”不等于“定稿”**——用户要求建立创意卡、作品卡或方案容器时，先判断其是在要求记录当前讨论，还是已批准中心判断和论证链。前者必须显式标为候选/讨论稿，并保留待定项；不得把 agent 的推断写成用户已确认的方向。

## 工作流程

### 1. 检索碎片与争点材料（必做）

- 本次 mine 新落的碎片（查 `01-碎片/` 最近变更）及其材料交接清单
- 既有碎片（按主题/标签搜 `01-碎片/` 和 `00-索引/碎片库.base`）
- 既有候选 idea（查 `02-创意/候选/` 做去重，避免重复产题）
- 材料交接中的来源锚点；若交接已过期或缺来源，重新核对，不把未核对的二手结论当事实

### 2. 比较争点，再挑骨架

先为每个核心争点并列：支持证据、反例、替代解释、适用边界/缺口、时间或制度语境；判断哪些解释确实竞争、哪些只是同一结论的补充。再从碎片和材料中挑作骨架的：
- **时效性**：最近刚提出的判断、有当下事件钩子的
- **主体性**：最强的、最能承重的主判断
- **张力**：材料之间能形成对立、递进或互相校准关系的

骨架碎片 = idea 的主干；支撑碎片 = 丰富论证；没用上的 = 留 `01-碎片/` 供以后。候选方向较多、融合质量拿不准或需要判断“现在值不值得写”时，按需读取 `references/选题判断.md`；不要把其中的辅助检查变成每次必跑的打分仪式。

### 3. 融合成论证组合候选

按 `02-创意/README.md` 的选题卡纪律，为每条可成立的组合产候选 idea。每个候选必须包含：
- **题目**：可独立成立的选题
- **中心判断**：一句话讲清这篇最终要让什么判断成立
- **论证链**：读者按什么顺序从材料走到中心判断
- **骨架碎片与支撑材料**：作主干的 1-3 条碎片（`[[]]` 引用）及关键来源锚点
- **使用的张力**：哪些反例、替代解释或语境会融入正文，如何参与论证
- **暂不采用/需正面处理的解释**：为什么不作为本篇主线，以及何时应在正文中正面交锋
- **钩子与时效/主体性说明**：具体场景/冲突/反常识，以及为什么当下值得写

不确定融合质量的，照样产出候选并交给 `507-grill`——agent 在候选里标注“这条组合我拿不准，X 和 Y 的关系可能勉强”。

### 4. 落 `02-创意/候选/` 交审

候选 idea 落进 `02-创意/候选/`（不是直接进 `02-创意/` 正式区），再进入 `507-grill` 对齐用户拥有的内容判断。若用户只确认“先建立容器”而未确认中心判断、论证链或作品流程，仍视为候选：文件必须标明已确认共识、讨论稿内容和待定项，不能以 `drafting（草拟）` 字样掩盖事实上被写死的方案。

通过 `507-grill` 审核用户拥有的内容判断：
- 在 2～5 条可竞争组合中，哪条最值得写 → 提升进 `02-创意/` 正式区
- 哪些组合的中心判断或论证链勉强 → 退回调整或放弃
- 哪些材料不足以支持候选 → 回到 mine 定向补材料后重融

### 5. 维护选题图谱

每个候选 idea 落盘后，同步维护 `00-索引/选题图谱.canvas`（加节点 + 受控词表边），按 `02-创意/README.md` 的选题流水线纪律。

### 6. 不抢下游

fuse 产出候选 idea 后停。经 `507-grill` 确认中心判断并进入正式区后，Agent 根据用户已经表达的交付目标路由：书面长文进入 forge，现场讲述进入 stage；若用户要的是面向人的方案，应先由 grill 把想法收敛成需求，再进入 frame。Agent 用一句话解释理由，不让用户选择 skill 名称。

## 和其他 skill 的边界

| skill | 分工 |
|---|---|
| **mine** | 素材 → 碎片（fuse 的上游） |
| **fuse** | 碎片 → 候选 idea（本 skill） |
| **forge** | idea → 长文（fuse 的下游） |
| **cast** | 碎片 → 知识库主题页（和 fuse 平行，都从碎片聚合，产物不同） |
| **stage** | idea → 讲稿（fuse 下游，允许更多 idea+碎片组合） |

**fuse vs cast 的关键区别**：都从碎片聚合，但产物不同——fuse 聚合成"值得写一篇的判断"（候选 idea，进 `02-创意/候选/`），cast 聚合成"一个主题的系统化主题页"（进 `04-知识库/`）。fuse 验收"够不够写一篇"，cast 验收"够不够成一个二次学习入口"。

## 启动姿势

当用户：
- mine 挖完一批碎片后说"看看能不能成题"
- 抛几个已有碎片问"这几个组合够不够写"
- 说"融一下""组个选题""挑骨架"

你就：
1. 检索 `01-碎片/`（本次新 + 既有）、mine 的材料交接和 `02-创意/候选/`（去重）。
2. 按核心争点比较多角色材料，再挑时效性/主体性碎片作骨架。
3. 有真实竞争解释时融合成 2～5 条候选 idea；只有一条时说明收敛理由。
4. 落 `02-创意/候选/` + 维护选题图谱；存在真实选择时携带候选进入 `507-grill`。
5. 停在候选或已确认正式创意，不抢 forge/stage/frame。

## 红线

- 不跳过检索直接产 idea（会漏掉既有碎片，违背"默认融合"）。
- 不硬凑融合（没有合适碎片就独立，不嫁接）。
- 不把本次碎片全塞进一个 idea（挑骨架，没用上的留着）。
- 不替用户拍板哪个 idea 成立（产候选后交给 `507-grill`）。
- 不写正文（归 forge）。
- 不和 mine / forge / cast 抢活。

## 完成与接力

- **完成信号**：可竞争解释已比较，候选 idea 已落盘并进入图谱；若材料只支持一条路径，收敛理由已写明。
- **产物**：`02-创意/候选/` 中的候选卡、来源锚点和选题图谱更新。
- **候选出口**：外部证据不足进入 `507-research`，碎片不足进入 `507-mine`；存在用户取舍进入 `507-grill`；中心判断确认后，书面作品进入 `507-forge`，现场内容进入 `507-stage`，面向人方案经 grill 收敛需求后进入 `507-frame`；只需候选池时直接结束。
- **回退条件**：材料不能形成独立判断时退回碎片池继续生长，不硬造 idea 或伪分歧。
