# commit 与 release 按可逆性边界拆分为两个 skill

`507-commit` 收缩为只做本地提交；推送、版本号定档、打 tag 和发布渠道（npm、GitHub Release 等）交给新设的 `507-release`。

## 为什么

`commit` 和 `push`/发布属于不同等级的可逆性：本地提交可 `reset`/`amend`，基本无代价；而推到远端、发到 npm、建 GitHub Release 基本不可逆，撤回成本高（别人已 pull、包已公开）。把两者塞进一个 skill 会让安全边界模糊——原来 `507-commit` 反复强调"默认不 push"，正是在硬守这条线。

把 push/发布拆成独立的 `507-release` 后，可逆性边界变成 skill 之间的天然分界：`commit` 的职责在本地提交完成即结束，`release` 是另一道门，每个不可逆对外动作执行前都向用户显式确认。这也让版本号定档、tag、CHANGELOG 迁移、渠道探测这些发布专属流程被收敛进一个有门禁的 skill，而不是每次发布临时拼凑散落在 `commit` 出口、全局 Git 规范和口头指令里的规则。

## 考虑过的备选

让 `507-commit` 在用户授权后内部执行 push/发布（不新增 skill）。否决：破坏性等级不同却共用一个 skill，安全边界靠"默认不 push"一句话维持，容易在授权后模糊；且发布流程（版本号、渠道、确认门）有足够独立职责和完成信号，值得单独固化。
